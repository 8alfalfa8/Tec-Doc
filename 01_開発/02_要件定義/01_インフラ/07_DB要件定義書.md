# 📘 DB要件定義書（実務例）

以下に、**金融機関向け DB要件書（実務例）** を提示します。<br>
想定：インターネットバンキング基盤<br>
準拠：FISC安全対策基準、金融庁監督指針<br>
DB基盤例：Amazon Web Services上の Amazon RDS（Oracle/PostgreSQL想定）

※実務提出可能レベルの粒度です。

---

# 1. 文書概要

| 項目     | 内容           |
| ------ | ------------ |
| 文書名    | DB要件書        |
| 対象システム | インターネットバンキング |
| 対象環境   | 本番／DR／検証／開発  |
| 作成部門   | 基盤部・アプリ部     |
| 承認     | IT統括部長／CISO  |

---

# 2. 目的

本書は、対象システムで利用するデータベース基盤の可用性・性能・容量・セキュリティ・運用に関する要件を定義する。

---

# 3. DB基本方針

1. 高可用性構成（Multi-AZ）
2. データ完全性最優先
3. トランザクション整合性確保（ACID準拠）
4. バックアップ・リストア検証必須
5. アクセス最小権限

---

# 4. DB構成要件

## 4.1 論理構成

| 区分 | 構成                   |
| -- | -------------------- |
| 本番 | Multi-AZ（同期レプリケーション） |
| DR | 別リージョン非同期レプリケーション    |
| 検証 | 単一構成                 |
| 開発 | 単一構成                 |

---

## 4.2 バージョン

| 項目    | 内容                  |
| ----- | ------------------- |
| DB種別  | Oracle / PostgreSQL |
| バージョン | 最新安定版（LTS）          |
| 文字コード | UTF-8               |

---

# 5. 可用性要件

| 項目       | 要件       |
| -------- | -------- |
| SLA      | 99.99%以上 |
| RTO      | 1時間以内    |
| RPO      | 5分以内     |
| フェイルオーバー | 自動切替     |

* AZ障害時自動昇格
* DR切替年1回試験

---

# 6. 性能要件

| 項目     | 要件      |
| ------ | ------- |
| 最大TPS  | 5,000   |
| 同時接続数  | 10,000  |
| 平均応答時間 | 100ms以内 |
| CPU使用率 | 平均70%未満 |

* ピーク：月末営業日
* 性能試験にて保証

---

# 7. 容量要件

| 項目     | 初期    | 3年後   |
| ------ | ----- | ----- |
| データ容量  | 2TB   | 6TB   |
| インデックス | 500GB | 1.5TB |
| ログ領域   | 1TB   | 3TB   |

* 成長率：年30%想定
* 空き容量30%以上確保

---

# 8. バックアップ要件

| 区分       | 頻度   | 保持  |
| -------- | ---- | --- |
| フル       | 日次   | 30日 |
| 差分       | 1時間毎 | 7日  |
| スナップショット | 週次   | 4世代 |

* 別リージョン保管
* 暗号化必須
* 年1回リストア検証

---

# 9. セキュリティ要件

## 9.1 アクセス制御

* 個人ID付与
* 共用アカウント禁止
* 最小権限付与
* IP制限

## 9.2 暗号化

| 対象     | 方式       |
| ------ | -------- |
| 保存データ  | AES256   |
| 通信     | TLS1.2以上 |
| バックアップ | 暗号化必須    |

## 9.3 監査ログ

* 全管理操作記録
* 7年以上保存
* 改ざん防止

---

# 10. トランザクション要件

* ACID保証
* 自動コミット禁止（業務制御）
* ロールバック可能設計
* 長時間トランザクション禁止

---

# 11. パフォーマンスチューニング方針

* インデックス設計標準化
* 実行計画レビュー
* 統計情報定期更新
* バッチ時間帯分離

---

# 12. 運用要件

## 12.1 パッチ管理

| 区分 | 適用期限   |
| -- | ------ |
| 重大 | 72時間以内 |
| 高  | 2週間以内  |

## 12.2 メンテナンス

* 定期VACUUM/REORG
* 統計更新月次
* ログ整理

---

# 13. 監視要件

| 監視項目 | 閾値   |
| ---- | ---- |
| CPU  | 80%  |
| メモリ  | 75%  |
| ディスク | 75%  |
| 接続数  | 80%超 |

* 24時間監視
* 重大時30分以内一次対応

---

# 14. データ保持・削除要件

* 取引データ保存：7年以上
* ログ保存：7年以上
* 論理削除＋物理削除ルール明確化
* 削除履歴保持

---

# 15. 移行要件

* 無停止データ移行
* 並行稼働1か月
* ロールバック手順整備

---

# 16. リスク管理

| リスク       | 対策     |
| --------- | ------ |
| ロック競合     | SQL最適化 |
| デッドロック    | 設計レビュー |
| 容量逼迫      | 監視強化   |
| バックアップ未検証 | 定期復元試験 |

---

# 🔎 実務上の重要ポイント

✔ RTO/RPOは経営承認取得<br>
✔ バックアップは「取得」ではなく「復元確認」まで<br>
✔ DR環境を忘れない<br>
✔ 長時間トランザクション設計禁止<br>
✔ インデックス設計は性能要件とセットで合意
