# 📘 インフラ構成方針書（論理）（実務例）

以下に、**インフラ構成方針書（論理）（実務例）** を提示します。<br>
想定：金融機関向け／基幹系システム／オンプレ＋クラウドDR構成<br>
準拠観点：FISC安全対策基準、金融庁監督指針

---

# 1. 目的

本書は、基幹系業務システムのインフラ論理構成に関する基本方針を定義し、可用性・性能・拡張性・セキュリティ・監査適合性を確保することを目的とする。

---

# 2. システム概要

| 項目      | 内容             |
| ------- | -------------- |
| 想定ユーザー数 | 3,000名         |
| 同時接続数   | 最大1,200        |
| 日次取引件数  | 約200万件         |
| データ容量   | 12TB（年15%増加想定） |
| 可用性目標   | 99.95%以上       |
| RTO     | 4時間            |
| RPO     | 15分            |

---

# 3. 論理レイヤ構成

## 3.1 レイヤ構成

```
[外部接続]
    ↓
[DMZ層]
    ↓
[AP層]
    ↓
[DB層]
    ↓
[バックアップ／監視層]
```

---

## 3.2 各レイヤの役割

| レイヤ | 役割      | 論理方針     |
| --- | ------- | -------- |
| DMZ | 外部公開API | FW経由のみ通信 |
| AP  | 業務処理    | 水平分散構成   |
| DB  | データ管理   | クラスタ構成   |
| 監視  | 統合監視    | ログ集中管理   |
| 管理  | 運用保守    | 業務NWと分離  |

---

# 4. 可用性方針

* すべての主要機器は冗長構成
* 単一障害点（SPOF）排除
* AP層はActive-Active
* DB層はクラスタ構成（Active-Standby）
* ネットワーク二重化
* 電源二重化

---

# 5. セキュリティ方針

## 5.1 ネットワーク分離

| セグメント | 接続制御      |
| ----- | --------- |
| DMZ   | 外部公開のみ    |
| 業務    | 内部通信のみ    |
| DB    | APのみ接続可   |
| 管理    | VPN＋多要素認証 |

---

## 5.2 アクセス制御

* 統合認証基盤利用
* 最小権限原則
* 特権ID分離
* 操作ログ完全取得（7年以上保存）
* 管理者アクセスは踏み台経由

---

# 6. 性能設計方針

* ピーク負荷の130％で設計
* CPU利用率70％以下を目標
* ストレージIOPS余裕20％確保
* ボトルネック可視化

---

# 7. 拡張性方針

* AP層はスケールアウト可能
* DBは将来シャーディング対応可能設計
* ストレージ増設可能構成
* DRはクラウド側へ拡張可能

---

# 8. ログ管理方針

| 種別       | 保存期間 | 保管方式   |
| -------- | ---- | ------ |
| 取引ログ     | 7年以上 | WORM保管 |
| 操作ログ     | 7年以上 | 集中管理   |
| セキュリティログ | 5年以上 | SIEM連携 |
| OSログ     | 1年以上 | 集中管理   |

---

# 9. バックアップ方針

* 日次フルバックアップ
* 15分毎ログバックアップ
* 別拠点保管
* 年1回復元試験実施

---

# 10. DR構成（論理）

* 本番：オンプレミス
* DR：クラウド（例：Amazon Web Services）
* 非同期レプリケーション
* 切替手順書整備
* 年1回DR切替訓練

---

# 11. 監視方針

* 死活監視
* リソース監視
* ログ監視
* セキュリティイベント監視
* 24時間365日監視体制

---

# 12. 制約条件

* 既存勘定系接続は専用線利用
* 既存認証基盤を利用
* 既存NWアドレス体系継承

---

# 13. リスクと対応

| リスク    | 対策      |
| ------ | ------- |
| DB単一障害 | クラスタ化   |
| 容量逼迫   | 月次容量分析  |
| 権限過剰   | 四半期棚卸   |
| DR未検証  | 年次試験義務化 |

---

# 🔎 実務で重要な観点

* ✔ 数値（RTO/RPO/可用性）を明示
* ✔ 論理と物理を分離
* ✔ セグメント分離を明確化
* ✔ ログ保存年数を明文化
* ✔ DRは「訓練前提」で記載
