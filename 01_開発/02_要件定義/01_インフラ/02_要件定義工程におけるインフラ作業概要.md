# ◆ 要件定義工程におけるインフラ作業概要

要件定義工程におけるインフラ作業は、**後工程（基本設計・詳細設計）で手戻りを発生させないための「制約条件の確定」と「非機能要件の具体化」** が最大の目的です。<br>
アプリ要件と異なり、インフラは“見えにくいが全体を制約する基盤”であるため、初期段階の精度が極めて重要です。

以下、実務レベルで体系的に整理します。

---

## 1. 要件定義工程におけるインフラの位置づけ

### 目的

* システム基盤の方式・制約条件を確定する
* 非機能要件を定量化する
* セキュリティ・運用・監査観点を織り込む
* コスト・構成規模の概算を出す
* 技術的リスクを洗い出す

### インフラ要件の対象領域

| 領域     | 内容                     |
| ------ | ---------------------- |
| 基盤方式   | オンプレ / クラウド / ハイブリッド   |
| ネットワーク | セグメント設計方針、閉域網、FW方針     |
| サーバ    | OS、仮想化、コンテナ、Kubernetes |
| データベース | RDB種別、冗長化方式            |
| 可用性    | 冗長化レベル、SLA             |
| セキュリティ | 認証、暗号化、監査ログ            |
| 運用     | 監視、バックアップ、DR           |
| 性能     | トランザクション数、応答時間         |
| コスト    | 月額見積、初期費用              |

## 2. タスク一覧（網羅版）

### ① 現行調査（As-Is把握）

#### タスク

* 現行構成図の入手
* HWスペック確認
* 利用中ミドルウェア確認
* 運用体制確認
* 障害履歴分析
* 性能実績値確認

#### 方法

* ヒアリング
* 既存設計書レビュー
* 監視ツールログ確認

#### 注意点

* 「資料が正しい前提」で進めない
* 実測値と契約値は分ける

#### 成果物

* 現行インフラ構成図
* 現行課題一覧

---

### ② 将来構想（To-Be方針策定）

#### タスク

* クラウド利用有無決定（例：Amazon Web Services、Microsoft Azure）
* 可用性レベル決定
* マルチAZ/リージョン方針
* コンテナ採用有無（例：Amazon EKS）

#### 方法

* 非機能要件ヒアリング
* TCO比較
* 技術検証（PoC）

#### 注意点

* 流行技術で決めない
* 運用可能性を必ず確認

#### 成果物

* インフラ方式方針書
* 構成方針決定記録

---

### ③ 非機能要件定義（最重要）

#### 1. 可用性

* 稼働率（例：99.9% / 99.99%）
* RTO / RPO
* 冗長化レベル

#### 2. 性能

* 最大同時接続数
* 秒間トランザクション
* ピーク時間帯負荷

#### 3. 拡張性

* 将来3年後のデータ量予測
* 水平スケール可否

#### 4. セキュリティ

* IAM設計方針
* 暗号化方式
* 監査ログ保存期間

#### 5. 運用

* 監視ツール
* バックアップ保持期間
* パッチ適用方針

---

#### 方法

* 非機能要件テンプレート活用
* 定量値で合意
* SLAとの整合確認

#### 注意点

* 「ベストエフォート」はNG
* 数値化できないものは再ヒアリング

#### 成果物

* 非機能要件定義書

---

### ④ ネットワーク要件整理

#### タスク

* VPC設計方針
* サブネット分割
* インターネット接続可否
* 閉域網接続（例：Direct Connect）

#### 方法

* 通信フロー図作成
* セキュリティポリシー整理

#### 注意点

* 監査証跡取得可能性
* 東西トラフィックも考慮

#### 成果物

* 通信要件定義書
* NW論理構成図（ドラフト）

---

### ⑤ データベース要件

#### タスク

* DB種別決定（Oracle / PostgreSQL 等）
* 冗長構成（Active-Standby等）
* データ容量見積
* バックアップ方式

#### 注意点

* トランザクション整合性
* ライセンス制約

#### 成果物

* DB要件定義書
* データ容量試算表

---

### ⑥ セキュリティ要件整理

#### タスク

* 認証方式（SSO、AD連携）
* 権限分離
* WAF有無
* 暗号化方式
* ログ保管方針

#### 方法

* リスク分析
* 脅威モデリング

#### 注意点

* 開発環境も対象にする
* ログ保存期間は監査要件確認

#### 成果物

* セキュリティ要件一覧
* リスク一覧表

---

### ⑦ 運用設計前提整理

#### タスク

* 運用体制（24h? 平日日中?）
* 障害一次受け
* バックアップ確認手順
* DR訓練頻度

#### 成果物

* 運用要件定義書
* 運用体制図

---

### ⑧ コスト試算

#### タスク

* 月額概算
* 初期構築費
* ライセンス費
* 保守費

#### 方法

* クラウド見積ツール利用
* リザーブド/オンデマンド比較

#### 注意点

* 将来拡張時コストも提示
* データ転送料を忘れない

#### 成果物

* インフラ概算見積書

## 4. 典型的な失敗例

| 失敗        | 原因       |
| --------- | -------- |
| 本番だけ冗長化   | 非機能要件未整理 |
| コスト超過     | 通信量未試算   |
| DR未考慮     | 監査要件確認不足 |
| パフォーマンス不足 | ピーク想定甘い  |

## 6. 成功させるための重要ポイント

### ① 数値で合意する

曖昧ワード禁止（高性能、安定など）

### ② 運用者を必ず巻き込む

設計者だけで決めない

### ③ 監査観点を早期反映

金融・公共は特に重要

### ④ 将来拡張前提で設計

3年後負荷想定を入れる

### ⑤ アプリ要件と並走

インフラ単独で決めない
