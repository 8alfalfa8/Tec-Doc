# 📘 DR基本設計書（実務例）

本例ではクラウド基盤に Amazon Web Services を利用する構成を前提とします。

# 1. 目的

本書は、災害発生時におけるシステム継続運用を目的とし、
復旧目標、切替方式、構成、運用手順を定義する。

# 2. DR設計方針

## 2.1 想定災害

| 区分     | 内容       |
| ------ | -------- |
| 自然災害   | 地震・台風・洪水 |
| 大規模障害  | リージョン停止  |
| データ破損  | 論理障害     |
| サイバー攻撃 | ランサムウェア  |

---

## 2.2 目標値定義

| 指標          | 目標       |
| ----------- | -------- |
| RTO（復旧時間目標） | 2時間以内    |
| RPO（復旧時点目標） | 15分以内    |
| 可用性         | 99.95%以上 |

---

# 3. DR構成概要

## 3.1 本番構成

* プライマリリージョン：東京
* マルチAZ構成
* Auto Scaling構成
* DBマルチAZ

## 3.2 DR構成方式

DR方式：**Warm Standby方式**

| 項目    | 内容          |
| ----- | ----------- |
| リージョン | 大阪          |
| サーバ   | 最小台数起動      |
| DB    | リードレプリカ常時同期 |
| データ同期 | 非同期（15分以内）  |

---

# 4. DRアーキテクチャ

## 4.1 レイヤ別設計

### ① ネットワーク

* VPC分離
* サブネット分離
* ルートテーブル事前設定済

### ② アプリケーション

* AMI/イメージ保管
* Auto Scaling事前定義

### ③ データベース

* クロスリージョンレプリケーション
* 自動フェイルオーバー非採用（手動切替）

### ④ ストレージ

* オブジェクトストレージのクロスリージョン複製

# 5. データ保護設計

| ----- | --------- | ---- |
| DB    | 日次フル＋5分ログ | 35日  |
| ファイル  | 世代管理      | 90日  |
| ログ    | 集中管理      | 1年   |

# 6. 切替方式設計

## 6.1 切替判断基準

以下のいずれかを満たす場合：

* リージョン全停止
* 2時間以内復旧不可判断
* 経営判断

---

## 6.2 切替手順（概要）

1. 障害宣言
2. DR発動決定
3. DB昇格
4. アプリ起動
5. DNS切替
6. 動作確認
7. サービス再開

---

## 6.3 DNS切替

TTL：60秒
切替時間目安：5〜10分
