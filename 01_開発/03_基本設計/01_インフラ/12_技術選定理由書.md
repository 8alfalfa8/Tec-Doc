# 3. 基盤（クラウド）選定

## 3.1 候補

* Amazon Web Services
* Microsoft Azure
* Google Cloud

## 3.2 比較

| 評価項目    | AWS | Azure | GCP |
| ------- | --- | ----- | --- |
| 国内実績    | ◎   | ○     | △   |
| サービス成熟度 | ◎   | ○     | ○   |
| DR機能    | ◎   | ○     | ○   |
| 技術者確保   | ◎   | ○     | △   |
| コスト     | ○   | ○     | ○   |

## 3.3 採用理由

AWSを採用する理由：

* 国内導入実績が最多
* サービス成熟度が高い
* マルチAZ・マルチリージョン設計が容易
* 技術者確保が容易
* DR機能が豊富

# 4. アプリケーション構成選定

## 4.1 アーキテクチャ

選定：3層アーキテクチャ

理由：

* 責務分離が明確
* スケール分離可能
* 保守性向上

---

## 4.2 コンテナ採用可否

候補：

* VM構成
* コンテナ（Kubernetes）
* サーバレス

| 項目    | VM | コンテナ | サーバレス |
| ----- | -- | ---- | ----- |
| 運用負荷  | ○  | △    | ◎     |
| 柔軟性   | ○  | ◎    | △     |
| 学習コスト | 低  | 高    | 中     |

結論：
→ 初期フェーズはVM採用（安定性優先）
将来的にコンテナ移行可能な設計とする

# 7. データベース選定

候補：

* MySQL
* PostgreSQL
* Oracle

比較：

| 項目     | MySQL | PostgreSQL | Oracle |
| ------ | ----- | ---------- | ------ |
| コスト    | ◎     | ◎          | ×      |
| 拡張性    | ○     | ◎          | ◎      |
| 商用サポート | ○     | ○          | ◎      |

選定：PostgreSQL

理由：

* ACID準拠
* JSON対応
* 拡張性高
* コストバランス良

# 8. 監視基盤選定

選定：

* クラウド標準監視＋外形監視ツール

理由：

* 導入容易
* API連携可能
* アラート自動化可能

# 9. セキュリティ技術選定

| IDS/IPS | クラウド標準   |

# 10. DR方式選定

候補：

* Backup型
* Warm Standby
* Active-Active

選定：Warm Standby

理由：

* RTO/RPO要件を満たす
* コストバランス良
* 運用複雑性低

| コスト増   | 予算超過   | 予約利用   |
