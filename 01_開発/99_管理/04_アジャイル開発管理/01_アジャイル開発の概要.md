# ◆ アジャイル開発

以下では、**アジャイル開発**（**Scrumを中心**）における作業を、
**①全体構造 → ②作業管理 → ③各工程・タスクの詳細（入力・手順・方法・成果物） → ④役割別視点 → ⑤実務上の注意点**
という流れで、**実務レベル**まで掘り下げて説明します。

---

## ① アジャイル開発の全体構造（俯瞰）

アジャイル開発は「工程」ではなく**反復（イテレーション）**で進みます。

```
プロダクトビジョン
  ↓
プロダクトバックログ
  ↓
スプリント（1〜4週間）
  ├ 計画
  ├ 設計・実装・テスト
  ├ レビュー
  └ 振り返り
```

特徴

* **価値単位（ユーザーストーリー）で進行**
* **計画 → 実行 → 検証 → 改善**を高速で回す
* 文書より **動くソフトウェア**

---

## ② 作業管理（アジャイル特有）

### 1. 管理単位

| 管理対象  | 内容               |
| ----- | ---------------- |
| プロダクト | プロダクトバックログ       |
| 期間    | スプリント（固定）        |
| 作業    | ユーザーストーリー / タスク  |
| 進捗    | 完了 / 未完了（％管理しない） |

### 2. 管理ツール（例）

* Jira / Azure DevOps / Backlog
* カンバンボード（ToDo / In Progress / Done）
* バーンダウンチャート

---

## ③ 各作業・タスクの詳細

### 1️⃣ プロダクトバックログ作成・管理

#### 作業内容

* 要求を**ユーザーストーリー**として整理
* 優先度付け・粒度調整

#### 入力情報

* ビジネス要求
* ユーザー課題
* KPI / OKR
* 技術制約

#### 作業手順

1. 要求を洗い出す
2. ユーザーストーリー化

   ```
   As a ○○
   I want △△
   So that □□
   ```
3. 受入条件（Acceptance Criteria）定義
4. 優先順位付け

#### 作業方法

* PO主導、チームレビュー
* INVEST原則（Independent, Negotiable, Valuable…）

#### 成果物

* プロダクトバックログ
* 各ストーリーの受入条件

---

### 2️⃣ バックログリファインメント（洗練）

#### 作業内容

* ストーリーを**実装可能レベル**まで具体化

#### 入力情報

* プロダクトバックログ
* 技術的知見

#### 作業手順

1. ストーリーの不明点洗い出し
2. 仕様・画面・API観点の整理
3. 見積り（ストーリーポイント）

#### 作業方法

* プランニングポーカー
* ホワイトボード設計

#### 成果物

* 見積済みバックログ
* 補足仕様メモ

---

### 3️⃣ スプリントプランニング

#### 作業内容

* 今スプリントで「何を作るか」を決定

#### 入力情報

* 優先度付きバックログ
* チームキャパシティ
* 前スプリント実績

#### 作業手順

1. スプリントゴール定義
2. ストーリー選択
3. タスク分解（設計/実装/テスト等）

#### 作業方法

* チーム全員参加
* WBS的に分解するが軽量

#### 成果物

* スプリントバックログ
* スプリントゴール

---

### 4️⃣ 設計・実装・テスト（スプリント内）

#### 共通管理

* **Definition of Done（DoD）**を満たすことが完了条件

---

#### A. 設計タスク

| 項目   | 内容                     |
| ---- | ---------------------- |
| 作業内容 | 画面/API/DB設計            |
| 入力   | ユーザーストーリー              |
| 方法   | 軽量設計（ホワイトボード、PlantUML） |
| 成果物  | 設計メモ、API定義             |

---

#### B. 実装タスク

| 項目   | 内容       |
| ---- | -------- |
| 作業内容 | コーディング   |
| 入力   | 設計情報     |
| 方法   | TDD/ペアプロ |
| 成果物  | ソースコード   |

---

#### C. テストタスク

| 項目   | 内容          |
| ---- | ----------- |
| 作業内容 | 単体/結合/受入テスト |
| 入力   | 実装物         |
| 方法   | 自動テスト、BDD   |
| 成果物  | テストコード、結果   |

---

### 5️⃣ デイリースクラム

#### 作業内容

* 進捗同期・障害共有

#### 入力情報

* 各自の作業状況

#### 手順

* 3点のみ共有

  1. 昨日やったこと
  2. 今日やること
  3. 困っていること

#### 成果物

* 課題リスト更新

---

### 6️⃣ スプリントレビュー

#### 作業内容

* 完成物を**ステークホルダーにデモ**

#### 入力情報

* Doneストーリー

#### 方法

* 実動デモ中心

#### 成果物

* フィードバック
* バックログ更新

---

### 7️⃣ レトロスペクティブ（振り返り）

#### 作業内容

* プロセス改善

#### 入力情報

* スプリント実績

#### 方法

* KPT（Keep / Problem / Try）
* 5 Whys

#### 成果物

* 改善アクション

---

## ④ 役割別の作業視点

| 役割  | 主な作業        |
| --- | ----------- |
| PO  | 価値最大化、優先度管理 |
| SM  | プロセス改善、障害除去 |
| Dev | 設計・実装・テスト   |
| QA  | 受入条件保証      |

---

## ⑤ 実務上の注意点（重要）

* **進捗％管理はしない**
* 完了定義（DoD）が曖昧だと破綻
* バックログは「常に変わる前提」
* ドキュメントは**必要最小限**

---

### まとめ（要点）

* アジャイルは「作業の束」ではなく**価値の流れ**
* 各タスクは
  **入力 → 軽量作業 → 明確なDone → 成果物**
  で構成される
* 管理の主役は「計画」ではなく「フィードバック」
