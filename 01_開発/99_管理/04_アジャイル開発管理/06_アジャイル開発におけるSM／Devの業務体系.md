# ◆ アジャイル開発におけるSM／Devの業務体系

以下では **Scrum／アジャイル体制を前提にした実務現場（特に日本のSIer・金融／公共案件で現実的に運用されている形）** を想定し、
**SM（Scrum Master）** と **開発チーム（Dev Team）** の業務体系を、**現場でそのまま使える粒度**で整理します。

---

## 1. SM（Scrum Master）の業務体系

---

### ① 役割の本質（Why）

**SMとは「Scrumが正しく機能し続けるための守護者であり、チーム改善の触媒」**

* 人・プロセス・組織の **詰まり（ボトルネック）を取り除く存在**
* POやDevを「管理」しない
* **自己組織化を成立させるための支援者**

> ❌ SM＝進捗管理者<br>
> ⭕ SM＝プロセスとチーム健全性の責任者

---

### ② 責任範囲（Accountability）

| 分類        | 内容           |
| --------- | ------------ |
| Scrumプロセス | イベントが正しく回ること |
| 障害除去      | チーム阻害要因の排除   |
| 改善推進      | 継続的改善の定着     |
| 教育        | Scrum理解促進    |
| 組織調整      | チーム外障害への対応   |

※ **成果物ではなく「状態」に責任を持つ**

---

### ③ タスク（What）

#### Scrumイベント運営

* スプリントプランニングの設計・進行
* デイリースクラムのファシリテーション
* スプリントレビュー支援
* レトロスペクティブ設計・促進

#### 改善・支援

* ボトルネック可視化
* ベロシティ・フロー分析
* 改善アクション定義・フォロー
* チームコーチング

#### 障害除去

* 外部依存調整
* 組織ルール／調整
* 環境・ツール課題解消

---

### ④ 入力（Input）

* チーム状況（発言・行動）
* メトリクス（ベロシティ、WIP等）
* レトロスペクティブ結果
* チームからの困りごと
* PO／組織からの制約

---

### ⑤ 作業方法・進め方（How）

* 観察 → 仮説 → 小さな改善
* 問いかけ中心（指示しない）
* 合意形成・心理的安全性確保
* 改善は **実験として扱う**

---

### ⑥ 注意事項（実務で効くポイント）

* 進捗管理役にならない
* POやDevの代弁者にならない
* 改善を押し付けない
* 形式Scrumに陥らない

---

### ⑦ 成果物（Deliverables）

* 改善アクションリスト
* チーム健全性向上
* 安定したスプリント運営
* 問題が表に出る文化

※ **文書より「状態」**

---

### ⑧ よくある失敗例（アンチパターン）

| パターン   | 内容      |
| ------ | ------- |
| 進捗係    | PM化     |
| 会議司会だけ | ファシリだけ  |
| 調整役過多  | チーム自律阻害 |
| 改善過剰   | 疲弊      |

---

## 2. 開発チーム（Dev Team）の業務体系

---

### ① 役割の本質（Why）

**Devチームとは「価値を“動くソフトウェア”として具現化する自己組織化チーム」**

* 指示待ちではなく **自律的に作る**
* 品質と技術的健全性に責任を持つ

> ❌ Dev＝実装要員<br>
> ⭕ Dev＝価値創出の主体

---

### ② 責任範囲（Accountability）

| 分類      | 内容        |
| ------- | --------- |
| 実装      | 設計・開発・テスト |
| 品質      | バグ・技術的負債  |
| 技術判断    | 実装方法・構成   |
| スプリント達成 | 目標達成      |
| 改善      | 開発プロセス改善  |

---

### ③ タスク（What）

#### スプリント内

* 設計（アーキ・詳細）
* 実装（コード）
* テスト（単体／結合）
* レビュー
* バグ修正

#### 継続活動

* リファクタリング
* 技術的負債解消
* 自動化（CI/CD）
* ドキュメント最小限整備

---

### ④ 入力（Input）

* Readyなプロダクトバックログ
* 受入条件
* 技術制約・非機能要件
* スプリントゴール

---

### ⑤ 作業方法・進め方（How）

* 自律的タスク分解
* ペア／モブ開発
* 小さく作って頻繁に統合
* テストファースト・CI

---

### ⑥ 注意事項（実務で効くポイント）

* 要件を勝手に変えない
* 技術最適を目的化しない
* ベロシティ至上主義に陥らない
* 「動く＝Done」としない

---

### ⑦ 成果物（Deliverables）

* 動作するインクリメント
* テスト結果
* 技術的健全性
* ナレッジ共有

---

### ⑧ よくある失敗例（アンチパターン）

| パターン  | 内容      |
| ----- | ------- |
| 指示待ち  | 自己組織化崩壊 |
| 品質後回し | 技術的負債増大 |
| 技術趣味化 | 価値不一致   |
| POA依存 | 仕様理解不足  |

---

## 3. SM × Dev の関係性（要点）

| 観点 | SM     | Dev    |
| -- | ------ | ------ |
| 目的 | チーム健全性 | 価値実装   |
| 指示 | しない    | 受けない   |
| 改善 | 促進     | 実行     |
| 成果 | 状態     | ソフトウェア |

---

### 現場の合言葉

> **「SMは環境を整え、Devは価値を作る」**
