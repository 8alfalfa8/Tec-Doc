# ◆ 大規模アジャイル（SAFe）開発

以下では、**大規模アジャイル（SAFe：Scaled Agile Framework）**を中心に、
**①全体構造 → ②レイヤー別作業構造 → ③各作業の詳細（入力・タスク・成果物） → ④作業管理・指標 → ⑤実務上の勘所**
という順で、**実案件（数十〜数百名規模）で使える粒度**で整理します。

---

## ① 大規模アジャイル（SAFe）の全体像

SAFeは「Scrumを拡張したもの」ではなく、
**企業レベルでアジャイルを成立させるための運営フレームワーク**です。

```
戦略・投資判断
   ↓
ポートフォリオ（予算・テーマ）
   ↓
バリューストリーム
   ↓
ART（複数チーム）
   ↓
チーム（Scrum/Kanban）
```

---

## ② SAFeのレイヤー別作業構造

| レイヤー           | 主目的      | 主な成果物         |
| -------------- | -------- | ------------- |
| ポートフォリオ        | 投資判断     | 戦略テーマ、エピック    |
| Large Solution | 巨大システム統合 | ソリューションビジョン   |
| プログラム（ART）     | 複数チーム連携  | PI目標、機能       |
| チーム            | 実装       | ストーリー、インクリメント |

---

## ③ レイヤー別 作業詳細

---

### 1️⃣ ポートフォリオレベル

#### 目的

* **何に投資するかを決める**
* 年次固定予算 → **リーン予算**へ

---

#### 主な作業

##### (1) 戦略テーマ策定

| 項目  | 内容            |
| --- | ------------- |
| 入力  | 経営戦略、KPI      |
| 作業  | 優先度の高い変革テーマ定義 |
| 成果物 | 戦略テーマ         |

---

##### (2) エピック管理（Lean Epic）

| 項目  | 内容                 |
| --- | ------------------ |
| 入力  | ビジネス課題             |
| 作業  | Epic仮説立案           |
| 方法  | Lean Business Case |
| 成果物 | ポートフォリオエピック        |

**Lean Business Case構成**

* 仮説
* 期待効果
* MVP
* コスト/リスク

---

##### (3) ポートフォリオカンバン

```
Funnel → Review → Analysis → Backlog → Implementing → Done
```

---

### 2️⃣ Large Solution レベル（※該当する場合）

#### 対象

* 複数ART・外部ベンダーを跨ぐ巨大システム

#### 主な作業

* ソリューションビジョン定義
* サブシステム間インタフェース調整
* 統合計画

#### 成果物

* Solution Intent
* 統合ロードマップ

---

### 3️⃣ プログラムレベル（ART）

> **ART（Agile Release Train）**
> 5〜12チーム（50〜125人）単位

---

#### 主な作業

##### (1) PIプランニング（最重要イベント）

| 項目  | 内容                |
| --- | ----------------- |
| 入力  | 上位エピック、戦略テーマ      |
| 作業  | 8〜12週の計画          |
| 方法  | Big Room Planning |
| 成果物 | PI目標、PIロードマップ     |

**作業手順**

1. ビジネスコンテキスト共有
2. 機能（Feature）説明
3. チーム別計画
4. 依存関係調整
5. リスクROAM

---

##### (2) 機能（Feature）管理

| 項目  | 内容         |
| --- | ---------- |
| 入力  | エピック       |
| 作業  | 機能分割       |
| 方法  | WSJF       |
| 成果物 | プログラムバックログ |

---

##### (3) システムデモ

* 全チーム統合成果のデモ
* 実動のみ（資料NG）

---

##### (4) Inspect & Adapt

| 内容   | 詳細      |
| ---- | ------- |
| 定量分析 | KPI・品質  |
| 定性分析 | 問題抽出    |
| 改善   | アクション定義 |

---

### 4️⃣ チームレベル（Scrum / Kanban）

#### 主な作業

| 作業      | 内容        |
| ------- | --------- |
| スプリント計画 | ストーリー選択   |
| 実装      | 設計・実装・テスト |
| デイリー    | 同期        |
| レビュー    | 検証        |
| レトロ     | 改善        |

---

## ④ 作業管理・指標（SAFe特有）

### 1. 管理対象

| 対象      | 指標      |
| ------- | ------- |
| ポートフォリオ | 投資対効果   |
| ART     | PI予測達成率 |
| チーム     | ベロシティ   |
| 品質      | 欠陥流出率   |

---

### 2. 代表的KPI

* Predictability Measure
* Feature Cycle Time
* Defect Density
* Business Value達成度

---

## ⑤ 実務で失敗しやすいポイント

#### ❌ よくある失敗

* SAFe＝会議増加
* PI計画がWBS化
* PO権限が弱い
* 依存関係放置

#### ✅ 成功の勘所

* **PI計画は「約束」ではなく「予測」**
* 依存関係は可視化最優先
* エピックは仮説検証
* ARTを「組織」ではなく「価値流れ」と見る

---

## ⑥ まとめ（構造理解）

```
経営判断（Why）
   ↓
エピック（What）
   ↓
Feature（How much）
   ↓
Story（How）
   ↓
Increment（Value）
```
