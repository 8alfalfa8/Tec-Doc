# ◆ アジャイル開発におけるPO／POA業務体系

ここでは **Scrum／アジャイルを前提とした実務現場（特に日本企業・SIer・金融／公共案件）** において成立する形で、
**PO（Product Owner）／POA（Product Owner Assistant / Proxy PO）** の業務体系を、**実務粒度**で整理します。

---

## 1. PO（Product Owner）の業務体系

---

### ① 役割の本質（Why）

**POとは「プロダクトの価値に対して最終責任を負い、意思決定を行う唯一の存在」**

* 「正しいものを作っているか」を保証する責任者
* 技術や方法論ではなく **ビジネス価値・業務価値の最大化** が使命
* Scrumにおける **唯一の意思決定者（Single Wringable Neck）**

> ❌ PO＝要件を書く人<br>
> ⭕ PO＝「何を作るか」「何を作らないか」を決める人

---

### ② 責任範囲（Accountability）

| 分類    | 内容                  |
| ----- | ------------------- |
| 価値最大化 | ROI、業務効果、KPI達成      |
| 優先順位  | プロダクトバックログの順位       |
| 受入責任  | スプリント成果物の受入／却下      |
| 方針提示  | プロダクトビジョン・ロードマップ    |
| 調整    | 経営層／業務部門／外部ステークホルダー |

※ **責任は委譲不可（作業は委譲可）**

---

### ③ タスク（What）

#### 戦略・上流

* プロダクトビジョン定義
* 価値仮説の明確化
* KPI・成功指標設定
* ロードマップ策定

#### バックログ管理

* プロダクトバックログの優先順位付け
* Epicレベルでのスコープ決定
* トレードオフ判断（品質・スコープ・期限）

#### スプリント関連

* スプリントレビューでの受入判断
* フィードバック反映可否の判断

---

### ④ 入力（Input）

* 事業戦略／中期計画
* 業務課題・業務要件
* 利用者・顧客フィードバック
* 法規制・契約制約
* 開発チームからの技術的制約情報

---

### ⑤ 作業方法・進め方（How）

* **Why → What を語り、How は委ねる**
* 優先順位は数値・効果で説明
* 仕様詳細はPOA・開発チームに委譲
* 決断を先延ばしにしない

---

### ⑥ 注意事項（実務で効くポイント）

* 「みんなで決める」をしない
* 技術設計に踏み込みすぎない
* 受入基準を感覚で判断しない
* スプリント中に優先順位を変えない

---

### ⑦ 成果物（Deliverables）

* プロダクトビジョン
* ロードマップ
* 優先順位付きプロダクトバックログ
* 受入判断結果（Accept / Reject）

---

### ⑧ よくある失敗例（アンチパターン）

| パターン   | 内容           |
| ------ | ------------ |
| PO不在   | 承認だけで意思決定しない |
| 合意依存   | 全員合意待ちで遅延    |
| 仕様過剰   | 詳細設計レベルまで書く  |
| 優先順位崩壊 | 全部重要         |

---

## 2. POA（Product Owner Assistant / Proxy PO）の業務体系

※ **日本型Scrumにおける最重要ロール**

---

### ① 役割の本質（Why）

**POAとは「POの意思決定を支えるために、要件・情報・選択肢を整備する実務責任者」**

* 業務とシステムの「翻訳者」
* POが決断できる状態を作る参謀

> ❌ POA＝POの代わり<br>
> ⭕ POA＝POの判断精度を高める存在

---

### ② 責任範囲（Accountability）

| 分類      | 内容                  |
| ------- | ------------------- |
| 要件具体化   | Epic → Story → Task |
| バックログ整備 | 粒度・順序・依存整理          |
| 調整      | 業務部門／開発チーム          |
| 非機能整理   | 性能・セキュリティ・監査        |
| 判断支援    | POへの選択肢提示           |

※ **最終決定責任は持たない**

---

### ③ タスク（What）

#### 要件定義・整理

* ユーザーストーリー作成
* 業務フロー整理
* 画面遷移・UI整理
* 受入条件（Acceptance Criteria）定義

#### バックログ管理

* バックログリファインメント主導
* Definition of Ready 満たし込み
* 依存関係整理

#### 調整・支援

* 業務部門との要件すり合わせ
* 開発チームからの質問対応
* POへの判断材料整理

---

### ④ 入力（Input）

* POのビジョン・方針
* 現行業務資料
* 業務部門ヒアリング結果
* 法規制・非機能要件
* 開発チームからの技術的制約

---

### ⑤ 作業方法・進め方（How）

* **曖昧さを潰すが、作り込みすぎない**
* 業務用語をシステム仕様に翻訳
* 文章＋図（業務フロー・画面）
* 選択肢と影響をセットでPOに提示

---

### ⑥ 注意事項（実務で効くポイント）

* POの代わりに決めない
* 開発側の代弁者にならない
* 要件を固定化しすぎない
* 「Ready」と「Done」を混同しない

---

### ⑦ 成果物（Deliverables）

* ユーザーストーリー
* 受入条件
* 業務フロー図
* 画面仕様簡易資料
* 整理済バックログ

---

### ⑧ よくある失敗例（アンチパターン）

| パターン  | 内容             |
| ----- | -------------- |
| 代理PO化 | 独断で仕様・優先順位決定   |
| 要件過多  | ウォーターフォール化     |
| 板挟み   | POと開発の緩衝材になり疲弊 |
| 曖昧放置  | 開発に丸投げ         |

---

## 3. PO／POAの境界線（最重要まとめ）

| 項目     | PO | POA   |
| ------ | -- | ----- |
| 価値責任   | ◎  | ×     |
| 優先順位決定 | ◎  | △（提案） |
| 要件詳細   | △  | ◎     |
| 受入判断   | ◎  | △（支援） |
| 実装方法   | ×  | ×     |

---

### 現場での合言葉

> **「POは決める、POAは整える」**
