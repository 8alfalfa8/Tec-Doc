# ◆ アジャイル開発におけるPO／POA業務体系

ここでは **Scrum／アジャイルを前提とした実務現場（特に日本企業・SIer・金融／公共案件）** において成立する形で、
**PO（Product Owner）／POA（Product Owner Assistant / Proxy PO）** の業務体系を、**実務粒度**で整理します。

---

## 0. PO（POA）の業務体系（全体像）

```

PO（POA）戦略
  ├─ プロダクトビジョン（PO）
  │     ├─ プロダクトビジョン・戦略策定（PO決定）
  │     ├─ プロダクトバックログの最終責任（PO責任）
  │     ├─ ステークホルダー調整・意思決定（PO決定）
  │     ├─ スプリント運営・価値検証（PO評価）
  │     └─ 受入・リリース判断（PO承認）
  │
  └─ プロダクト運営実務（POA）
        ├─ バックログ管理（POA主導）
        ├─ 要件具体化（POA主導）
        ├─ ステークホルダー調整（POA代行）
        ├─ データ分析・意思決定支援（POA主導）
        └─ スクラムイベント成立支援（POA主導）

```

## 1. PO（Product Owner）の業務体系

---

### ① 役割の本質（Why）

> **プロダクトオーナー（PO）とは、プロダクトの価値を最大化することに全責任を負い、
> 「何を作るか／作らないか」を最終決定する唯一の責任者である。**
> ユーザー・ビジネス・リスクの観点からプロダクトビジョンを定義し、
> プロダクトバックログの内容と優先順位を通じて、その意思を開発チームに明確に伝える。
> スプリントごとの成果を検証し、継続的に価値判断を更新する役割を担う。

**POとは「プロダクトの価値に対して最終責任を負い、意思決定を行う唯一の存在」**

* 「正しいものを作っているか」を保証する責任者
* 技術や方法論ではなく **ビジネス価値・業務価値の最大化** が使命
* Scrumにおける **唯一の意思決定者（Single Wringable Neck）**

> ❌ PO＝要件を書く人<br>
> ⭕ PO＝「何を作るか」「何を作らないか」を決める人

---

### ② 責任範囲（Accountability）

| 分類    | 内容                  |
| ----- | ------------------- |
| 価値最大化 | ROI、業務効果、KPI達成      |
| 優先順位  | プロダクトバックログの順位       |
| 受入責任  | スプリント成果物の受入／却下      |
| 方針提示  | プロダクトビジョン・ロードマップ    |
| 調整    | 経営層／業務部門／外部ステークホルダー |

※ **責任は委譲不可（作業は委譲可）**

---

### ③ POの具体業務　タスク体系（What）

#### ③-1 プロダクトビジョン・戦略策定・上流

##### タスク

* プロダクトビジョン定義
* 価値仮説の明確化
* KPI・成功指標設定
* ロードマップ策定

##### 入力

* 事業戦略
* 市場調査・顧客要望
* 制約条件（予算・期限・法規）

##### 作業内容

* 誰のどんな課題を解くかを明文化
* 成功状態（KPI/OKR）を定義
* 価値仮説を言語化

##### 手順

1. ターゲットユーザー定義
2. 課題仮説整理
3. 解決価値の明文化
4. 成功指標定義
5. ステークホルダー共有

##### 注意事項

* 解決手段を先に決めない
* 抽象的なスローガンにしない
* 継続的に見直す

##### 成果物

* プロダクトビジョン文書
* プロダクトゴール
* KPI/OKR一覧

#### ③-2 バックログ管理

##### タスク

* プロダクトバックログ承認
* プロダクトバックログの優先順位付け
* Epicレベルでのスコープ決定
* トレードオフ判断（品質・スコープ・期限）

##### 入力

* 要望一覧
* 技術的制約
* POA/Devからの提案

##### 作業内容

* バックログ内容の妥当性確認
* 優先順位の最終決定
* 「やらない」判断

##### 手順

1. 優先度案レビュー
2. 価値・リスク比較
3. 判断理由整理
4. 決定・共有

##### 注意事項

* 多数決にしない
* 技術都合に引きずられない
* 判断理由を説明可能にする

##### 成果物

* 承認済みプロダクトバックログ
* 優先順位決定理由メモ

#### ③-3 ステークホルダー調整・意思決定

##### タスク

* 合意形成
* 要望取捨選択
* 期待値調整

##### 入力

* ステークホルダー要望
* 制約条件
* 開発状況

##### 作業内容

* 要望背景整理
* 実現可否判断
* 最終決断

##### 手順

1. 要望ヒアリング
2. 背景・価値整理
3. 判断案検討
4. 決断
5. 説明

##### 注意事項

* 全員を満足させようとしない
* 決断を先送りしない
* 説明責任を果たす

##### 成果物

* 合意事項
* 判断理由説明資料

#### ③-4 スプリント運営・価値検証

##### タスク

* スプリントゴール承認
* レビューでの価値評価
* 次スプリント判断

##### 入力

* スプリント成果物
* KPI実績
* ユーザーフィードバック

##### 作業内容

* 成果物が価値を生んだか評価
* 次の優先度調整

##### 手順

1. 成果確認
2. KPI確認
3. 継続/変更判断
4. バックログ更新

##### 注意事項

* 完成度ではなく価値で判断
* 過去判断に固執しない

##### 成果物

* レビュー評価
* 更新済みバックログ



#### ③-5 受入・リリース判断

##### タスク

* 受入判断
* リリース可否判断

##### 入力

* 受入条件達成状況
* 品質レポート

##### 作業内容

* ビジネス観点での受入判断

##### 注意事項

* 技術詳細に踏み込みすぎない
* 品質責任をDevから奪わない

##### 成果物

* 受入承認
* リリース判断記録

---

### ④ PO業務作業全体一覧(入力 → 作業方法・進め方 → 注意事項 → 成果物 → 失敗例)

#### ④-1 入力（Input）

* 事業戦略／中期計画
* 業務課題・業務要件
* 利用者・顧客フィードバック
* 法規制・契約制約
* 開発チームからの技術的制約情報


#### ④-2 作業方法・進め方（How）

* **Why → What を語り、How は委ねる**
* 優先順位は数値・効果で説明
* 仕様詳細はPOA・開発チームに委譲
* 決断を先延ばしにしない

#### ④-3 注意事項（実務で効くポイント）

* 「みんなで決める」をしない
* 技術設計に踏み込みすぎない
* 受入基準を感覚で判断しない
* スプリント中に優先順位を変えない

#### ④-4 成果物（Deliverables）

* プロダクトビジョン
* ロードマップ
* 優先順位付きプロダクトバックログ
* 受入判断結果（Accept / Reject）


#### ④-5 よくある失敗例（アンチパターン）

| パターン   | 内容           |
| ------ | ------------ |
| PO不在   | 承認だけで意思決定しない |
| 合意依存   | 全員合意待ちで遅延    |
| 仕様過剰   | 詳細設計レベルまで書く  |
| 優先順位崩壊 | 全部重要         |

## 2. POA（Product Owner Assistant / Proxy PO）の業務体系

※ **日本型Scrumにおける最重要ロール**

---

### ① 役割の本質（Why）

**POAとは「POの意思決定を支えるために、要件・情報・選択肢を整備する実務責任者」**

* 業務とシステムの「翻訳者」
* POが決断できる状態を作る参謀

> ❌ POA＝POの代わり<br>
> ⭕ POA＝POの判断精度を高める存在

---

### ② 責任範囲（Accountability）

| 分類      | 内容                  |
| ------- | ------------------- |
| 要件具体化   | Epic → Story → Task |
| バックログ整備 | 粒度・順序・依存整理          |
| 調整      | 業務部門／開発チーム          |
| 非機能整理   | 性能・セキュリティ・監査        |
| 判断支援    | POへの選択肢提示           |

※ **最終決定責任は持たない**

---

### ③ POAの具体業務　タスク体系（What）

#### ③-1 要件定義・整理（要件具体化・ストーリー化）

##### タスク
* ユーザーストーリー作成
* 業務フロー整理
* 画面遷移・UI整理
* 受入条件（Acceptance Criteria）定義
* 非機能要件整理

##### 入力

* ビジネス要件
* 業務フロー
* 法規・ガイドライン
* 既存システム仕様

##### 作業内容

* ユーザー視点でのストーリー化
* 受入条件（Given/When/Then）整理
* 非機能（性能・セキュリティ）明文化

##### 作業進め方

1. ペルソナ・利用シーン整理
2. ストーリー文作成
3. 受入条件を定義
4. 開発チームと認識合わせ
5. POレビュー

##### 注意事項

* 実装指示書にならないこと
* ACは検証可能な表現にする
* POの意図を歪めない

##### 成果物

* ユーザーストーリー
* 受入条件一覧
* 非機能要件メモ


#### ③-2 バックログ管理・優先順位付け

##### タスク

* バックログ精査・分解
* 優先度再評価
* バックログリファインメント主導
* Definition of Ready 満たし込み
* 依存関係整理

##### 入力

* プロダクトビジョン
* ビジネス要求（企画書・要望）
* KPI/OKR
* 技術的制約・工数見積

##### 作業内容

* エピック → ストーリー → タスクへの分解
* ビジネス価値・緊急度・リスクの整理
* 優先順位付け指標の作成（WSJFなど）

##### 作業進め方

1. 要求をエピック単位で整理
2. ストーリーに分解（INVEST原則）
3. 各ストーリーに価値・コスト・リスクを付与
4. 優先順位案を作成
5. POへ判断材料として提示

##### 注意事項

* POの判断を「誘導」しない
* 技術都合だけで優先度を決めない
* チームのベロシティを無視しない

##### 成果物

* プロダクトバックログ
* 優先度根拠資料
* リファインメント用説明資料

#### ③-3 調整・支援

##### タスク
* 要望ヒアリング
* 調整・交渉
* 合意形成支援
* 業務部門との要件すり合わせ
* 開発チームからの質問対応
* POへの判断材料整理

##### 入力

* ステークホルダー要望
* スプリント計画
* 制約条件

##### 作業内容

* 要望の背景・目的の整理
* 実現可否の整理
* 優先順位調整案の作成

##### 作業進め方

1. 要望ヒアリング
2. 目的と価値を言語化
3. 実現案・代替案整理
4. POへ報告
5. ステークホルダーへフィードバック

##### 注意事項

* POの代弁者であり代理決裁者ではない
* 感情論に引きずられない
* 記録を必ず残す

##### 成果物

* 調整メモ
* 合意事項一覧
* 要望→バックログ対応表

#### ③-4 POの意思決定支援（分析・提案）

##### タスク

* データ分析
* 仮説検証支援
* 投資対効果整理

##### 入力

* 利用データ
* アンケート結果
* KPI実績

##### 作業内容

* 課題仮説立案
* 数値分析
* 改善案提示

##### 手順

1. 現状データ収集
2. 仮説立案
3. 効果試算
4. 優先度案作成
5. POへ提案

##### 注意事項

* 結論を押し付けない
* 数字の前提条件を明示
* 小さく検証できる提案にする

##### 成果物

* 分析レポート
* 改善提案資料
* KPIトラッキング資料

---
### ④ POA業務作業全体一覧(入力 → 作業方法・進め方 → 注意事項 → 成果物 → 失敗例)

#### ④-1 入力（Input）

* POのビジョン・方針
* 現行業務資料
* 業務部門ヒアリング結果
* 法規制・非機能要件
* 開発チームからの技術的制約

---

#### ④-2 作業方法・進め方（How）

* **曖昧さを潰すが、作り込みすぎない**
* 業務用語をシステム仕様に翻訳
* 文章＋図（業務フロー・画面）
* 選択肢と影響をセットでPOに提示

---

#### ④-3 注意事項（実務で効くポイント）

* POの代わりに決めない
* 開発側の代弁者にならない
* 要件を固定化しすぎない
* 「Ready」と「Done」を混同しない

---

#### ④-4 成果物（Deliverables）

* ユーザーストーリー
* 受入条件
* 業務フロー図
* 画面仕様簡易資料
* 整理済バックログ

---

#### ④-5 よくある失敗例（アンチパターン）

| パターン  | 内容             |
| ----- | -------------- |
| 代理PO化 | 独断で仕様・優先順位決定   |
| 要件過多  | ウォーターフォール化     |
| 板挟み   | POと開発の緩衝材になり疲弊 |
| 曖昧放置  | 開発に丸投げ         |

## 3. PO／POAの境界線（最重要まとめ）

| 項目     | PO | POA   |
| ------ | -- | ----- |
| 価値責任   | ◎  | ×     |
| 優先順位決定 | ◎  | △（提案） |
| 要件詳細   | △  | ◎     |
| 受入判断   | ◎  | △（支援） |
| 実装方法   | ×  | ×     |

### 現場での合言葉

> **「POは決める、POAは整える」**

---
