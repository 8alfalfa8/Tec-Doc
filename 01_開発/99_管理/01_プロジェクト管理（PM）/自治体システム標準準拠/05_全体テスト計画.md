# ◆ 全体テスト計画（自治体システム標準準拠）

以下では、**自治体システム標準準拠（デジタル庁標準仕様準拠）における「全体テスト計画」作業**について、
**PL／PM支援・レビュー・決裁対応**まで含めた実務レベルで詳しく整理します。

---

## 1. 全体テスト計画の位置づけ（自治体標準特有）

### ① 一般的な全体テスト計画との違い

| 観点    | 一般業務システム | 自治体標準準拠システム                |
| ----- | -------- | -------------------------- |
| 主目的   | 品質確保     | **標準準拠・法令遵守・業務継続性の証明**     |
| テスト範囲 | 機能中心     | **機能＋標準仕様＋外部連携＋移行＋運用**     |
| 利用者   | ユーザ部門    | **自治体（業務・情シス）＋監査＋デジタル庁対応** |
| 成果物   | テスト計画書   | **決裁可能な公式文書**              |

👉 **「品質保証」ではなく「説明責任（アカウンタビリティ）」が主目的**

---

## 2. 全体テスト計画作業の全体像

```
要件定義完了
   ↓
基本設計
   ↓
【全体テスト計画】← 本作業（横断）
   ↓
各工程テスト設計
   ↓
テスト実施・結果報告
   ↓
本番移行／稼働判定
```

## 4. テスト方針策定（最重要）

### ① 基本方針（例）

* デジタル庁「標準仕様書」に準拠していることを**テストで証明**
* 住民サービス影響がある機能は**重点テスト**
* 移行・切替・障害時対応を**本番想定で確認**
* テスト結果は**自治体決裁に耐える形式で記録**

👉 **「動いた」ではなく「説明できる」ことが必須**

---

## 5. テスト範囲定義（自治体標準特有）

### ① 必須テスト範囲

| 区分        | 内容          |
| --------- | ----------- |
| 機能テスト     | 標準業務・独自業務   |
| 標準準拠テスト   | 標準仕様との差異検証  |
| 外部連携テスト   | 国・都道府県・他自治体 |
| データ移行テスト  | 既存データ完全性    |
| 運用テスト     | 運用手順・障害対応   |
| セキュリティテスト | 個人情報・認証     |
| 災害・BCPテスト | 業務継続性       |

---

## 6. テスト工程構成（典型）

### ① 工程別整理

| 工程      | 目的          |
| ------- | ----------- |
| 単体テスト   | プログラム単位の妥当性 |
| 結合テスト   | 機能間連携       |
| 総合テスト   | 業務シナリオ確認    |
| 標準準拠テスト | 標準仕様適合確認    |
| 運用テスト   | 実運用確認       |
| 受入テスト   | 自治体最終確認     |

※ **自治体標準では「標準準拠テスト」を独立させるのが基本**

---

## 7. 標準準拠テストの設計方針

### ① 観点例

| 観点    | 内容        |
| ----- | --------- |
| 機能有無  | 標準必須機能の実装 |
| 仕様差異  | 標準との差分    |
| 入出力   | 標準帳票・CSV  |
| コード体系 | 標準コード使用   |
| 画面項目  | 標準項目遵守    |

👉 **Fit & Gap一覧とテスト観点を必ず紐づける**

---

## 8. 外部連携・移行テスト計画

### ① 外部連携テスト

* 国システム（J-LIS等）とのIF確認
* ファイル／API形式
* 異常系（タイムアウト・不正値）

### ② データ移行テスト

| 観点    | 内容      |
| ----- | ------- |
| 件数一致  | 移行前後比較  |
| 内容一致  | 金額・日付   |
| 業務影響  | 移行後業務確認 |
| リハーサル | 本番想定    |

---

## 9. 非機能テスト計画（公共向け）

### ① 必須観点

| 分類     | 内容        |
| ------ | --------- |
| 性能     | ピーク時応答    |
| 可用性    | 冗長切替      |
| セキュリティ | 権限・ログ     |
| 運用     | 監視・バックアップ |
| 災害     | BCP訓練     |

---

## 10. 体制・役割定義

### ① 典型体制

| 役割    | 主な責任  |
| ----- | ----- |
| 自治体   | 受入・判定 |
| PL/PM | 統括・判断 |
| SIer  | テスト実施 |
| 運用部門  | 運用確認  |

👉 **「誰がOKと言うのか」を明確にする**

---

## 11. 合否判定基準（決裁用）

### ① 例

* 重大障害：0件
* 軽微障害：是正計画提出
* 標準準拠：**全必須項目適合**

## 13. PL/PM支援の実務ポイント

* **標準仕様書 × テスト観点のトレーサビリティ**
* SIer任せにしない
* 決裁文書としての表現（曖昧語NG）
* 後年の監査を想定
