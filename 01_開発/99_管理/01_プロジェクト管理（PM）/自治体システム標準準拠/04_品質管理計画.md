# ◆ 品質管理計画（自治体システム標準準拠）

以下は、**自治体システム標準準拠（デジタル庁標準仕様・ガバメントクラウド前提）における「品質管理計画」作業**を、
**PL／PM支援・監査耐性・ベンダー統制**の観点で、実務レベルまで落として詳しく説明したものです。

---

## 1. 品質管理計画の位置づけ（なぜ重要か）

自治体システム標準準拠における品質管理計画は、単なる「開発品質」ではなく、次を同時に満たす必要があります。

#### 標準準拠案件特有の要求

* デジタル庁「標準仕様書」への**適合性の担保**
* 将来の**国・自治体監査への説明責任**
* ベンダー交代・横断展開を前提とした**再現性・透明性**
* 業務・法令・制度改正に耐える**変更耐性**

👉 そのため、
**品質管理計画 ＝ 技術品質 × 業務品質 × 標準準拠品質 × 証跡品質**
を統合した計画となります。

---

## 2. 品質管理計画の全体構成（標準）

自治体標準準拠案件での品質管理計画書は、概ね以下の章立てになります。

```
1. 目的・適用範囲
2. 品質方針・品質目標
3. 品質管理体制（役割・責任）
4. 管理対象成果物と品質基準
5. 工程別品質管理方法
6. レビュー計画
7. テスト・検証計画との連携
8. 不具合管理・是正管理
9. 標準仕様適合性管理
10. 変更管理・構成管理との関係
11. 品質評価・報告方法
12. 監査・検査対応
```

---

## 3. 作業① 品質方針・品質目標の策定

### ① 品質方針の定義

自治体向けでは、以下を明示します。

* 標準仕様への**完全準拠を最優先**
* 業務影響・住民影響を最小化
* 証跡主義（説明可能性）の徹底

#### 記載例

> 本プロジェクトでは、デジタル庁が定める標準仕様書に完全準拠することを品質方針とし、
> 各工程において成果物の妥当性・完全性・追跡性を確保する。

---

### ② 品質目標（KPI）の設定

定量指標を設定します。

| 分類   | 指標例          |
| ---- | ------------ |
| レビュー | 指摘是正率 100%   |
| 不具合  | 本番障害 0件      |
| 標準準拠 | 適合率 100%     |
| 文書   | 承認成果物未承認率 0% |

---

## 4. 作業② 品質管理体制の設計

### 役割定義（例）

| 役割       | 主な責任           |
| -------- | -------------- |
| 発注者（自治体） | 最終承認、品質判断      |
| PL/PM支援  | 品質統制・第三者視点レビュー |
| SIer PM  | 実装品質の確保        |
| QA担当     | レビュー運営、不具合管理   |

👉 **PL/PM支援は「品質の番人」**として明確に位置づけるのが重要です。

---

## 5. 作業③ 管理対象成果物と品質基準の定義

### 管理対象成果物（例）

| 工程   | 成果物           |
| ---- | ------------- |
| 要件定義 | 要件定義書、Fit&Gap |
| 設計   | 基本設計書、詳細設計書   |
| 開発   | ソースコード、設定定義   |
| テスト  | 試験計画書、結果報告書   |
| 移行   | 移行計画書、検証結果    |

---

### 品質基準（例）

| 観点   | 基準         |
| ---- | ---------- |
| 完全性  | 必須項目漏れなし   |
| 妥当性  | 業務要件と一致    |
| 一貫性  | 他成果物と矛盾なし  |
| 標準準拠 | 標準仕様との乖離なし |
| 証跡性  | 根拠資料明示     |

---

## 6. 作業④ 工程別品質管理方法

### 要件定義工程

* 標準仕様トレーサビリティ管理
* Fit & Gap妥当性確認
* 非機能要件（性能・可用性・セキュリティ）重点レビュー

### 設計工程

* 要件トレーサビリティチェック
* 標準機能優先設計の確認
* 個別カスタマイズ抑制チェック

### 開発工程

* コーディング規約遵守
* 静的解析・セキュリティチェック

### テスト工程

* 要件ベース試験網羅性確認
* 標準仕様適合試験の実施

---

## 7. 作業⑤ レビュー計画の策定（重要）

### レビュー種別

| レビュー    | 主体      |
| ------- | ------- |
| 内部レビュー  | SIer    |
| 第三者レビュー | PL/PM支援 |
| 承認レビュー  | 自治体     |

### レビュー観点

* 標準仕様準拠
* 業務適合性
* 将来制度改正耐性
* 他団体横断利用可能性

---

## 8. 作業⑥ 不具合・是正管理

### 不具合管理ルール

* 重大度（Critical / High / Medium / Low）
* 原因分析（人・プロセス・設計）
* 再発防止策の明示

👉 **是正完了＝再発防止まで含めて完了**

---

## 9. 作業⑦ 標準仕様適合性管理（自治体特有）

### 適合性管理方法

* 標準仕様項番ごとの対応表作成
* 未対応・代替対応の理由明示
* デジタル庁QA・補足資料の反映

### 成果物

* 標準仕様適合性一覧
* 差分・例外管理表

---

## 10. 作業⑧ 品質評価・報告

### 定期報告内容

* 品質KPI達成状況
* 不具合件数推移
* 標準準拠状況

👉 **自治体決裁向けに「一目で分かる」資料が必須**

---

## 11. 作業⑨ 監査・検査対応

* 監査観点一覧の事前整理
* 成果物・証跡の即時提示可能化
* 口頭説明用シナリオ整備

---

## 12. PL/PM支援としての実務ポイント

* 「品質＝SIer任せ」にしない
* 文書品質・説明品質を重視
* 標準準拠を**設計思想レベル**で統制
* 将来ベンダー交代を見据える

---

### まとめ（重要）

自治体システム標準準拠における品質管理計画とは、

> **「作ったものが正しい」ではなく
> 「なぜ正しいかをいつでも説明できる状態を作る計画」**

です。
