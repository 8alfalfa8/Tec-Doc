# ◆ 品質管理計画（自治体システム標準準拠）

以下は、**自治体システム標準準拠（デジタル庁標準仕様・ガバメントクラウド前提）における「品質管理計画」作業**を、
**PL／PM支援・監査耐性・ベンダー統制**の観点で、実務レベルまで落として詳しく説明したものです。

---

## 1. 品質管理計画の位置づけ（なぜ重要か）

自治体システム標準準拠における品質管理計画は、単なる「開発品質」ではなく、次を同時に満たす必要があります。

#### 標準準拠案件特有の要求

* デジタル庁「標準仕様書」への**適合性の担保**
* 将来の**国・自治体監査への説明責任**
* ベンダー交代・横断展開を前提とした**再現性・透明性**
* 業務・法令・制度改正に耐える**変更耐性**

👉 そのため、
**品質管理計画 ＝ 技術品質 × 業務品質 × 標準準拠品質 × 証跡品質**
を統合した計画となります。

---

## 2. 品質管理計画の全体構成（標準）

自治体標準準拠案件での品質管理計画書は、概ね以下の章立てになります。

```
1. 目的・適用範囲
2. 品質方針・品質目標
3. 品質管理体制（役割・責任）
4. 管理対象成果物と品質基準
5. 工程別品質管理方法
6. レビュー計画
7. テスト・検証計画との連携
8. 不具合管理・是正管理
9. 標準仕様適合性管理
10. 変更管理・構成管理との関係
11. 品質評価・報告方法
12. 監査・検査対応
```

---

## 3. 作業① 品質方針・品質目標の策定

### ① 品質方針の定義

自治体向けでは、以下を明示します。

* 標準仕様への**完全準拠を最優先**
* 業務影響・住民影響を最小化
* 証跡主義（説明可能性）の徹底

#### 記載例

> 本プロジェクトでは、デジタル庁が定める標準仕様書に完全準拠することを品質方針とし、
> 各工程において成果物の妥当性・完全性・追跡性を確保する。

---

### ② 品質目標（KPI）の設定

定量指標を設定します。

| 分類   | 指標例          |
| ---- | ------------ |
| レビュー | 指摘是正率 100%   |
| 不具合  | 本番障害 0件      |
| 標準準拠 | 適合率 100%     |
| 文書   | 承認成果物未承認率 0% |

---

## 4. 作業② 品質管理体制の設計

### 役割定義（例）

| 役割       | 主な責任           |
| -------- | -------------- |
| 発注者（自治体） | 最終承認、品質判断      |
| PL/PM支援  | 品質統制・第三者視点レビュー |
| SIer PM  | 実装品質の確保        |
| QA担当     | レビュー運営、不具合管理   |

👉 **PL/PM支援は**「**品質の番人**」として明確に位置づけるのが重要です。

---

## 5. 作業③ 管理対象成果物と品質基準の定義

### 管理対象成果物（例）

| 工程   | 成果物           |
| ---- | ------------- |
| 要件定義 | 要件定義書、Fit&Gap |
| 設計   | 基本設計書、詳細設計書   |
| 開発   | ソースコード、設定定義   |
| テスト  | 試験計画書、結果報告書   |
| 移行   | 移行計画書、検証結果    |

---

### 品質基準（例）

| 観点   | 基準         |
| ---- | ---------- |
| 完全性  | 必須項目漏れなし   |
| 妥当性  | 業務要件と一致    |
| 一貫性  | 他成果物と矛盾なし  |
| 標準準拠 | 標準仕様との乖離なし |
| 証跡性  | 根拠資料明示     |

---

## 6. 作業④ 工程別品質管理方法

### 要件定義工程

* 標準仕様トレーサビリティ管理
* Fit & Gap妥当性確認
* 非機能要件（性能・可用性・セキュリティ）重点レビュー

### 設計工程

* 要件トレーサビリティチェック
* 標準機能優先設計の確認
* 個別カスタマイズ抑制チェック

### 開発工程

* コーディング規約遵守
* 静的解析・セキュリティチェック

### テスト工程

* 要件ベース試験網羅性確認
* 標準仕様適合試験の実施

---

## 7. 作業⑤ レビュー計画の策定（重要）

### レビュー種別

| レビュー    | 主体      |
| ------- | ------- |
| 内部レビュー  | SIer    |
| 第三者レビュー | PL/PM支援 |
| 承認レビュー  | 自治体     |

### レビュー観点

* 標準仕様準拠
* 業務適合性
* 将来制度改正耐性
* 他団体横断利用可能性

---

## 8. 作業⑥ 不具合・是正管理

### 不具合管理ルール

* 重大度（Critical / High / Medium / Low）
* 原因分析（人・プロセス・設計）
* 再発防止策の明示

👉 **是正完了＝再発防止まで含めて完了**

---

## 9. 作業⑦ 標準仕様適合性管理（自治体特有）

### 適合性管理方法

* 標準仕様項番ごとの対応表作成
* 未対応・代替対応の理由明示
* デジタル庁QA・補足資料の反映

### 成果物

* 標準仕様適合性一覧
* 差分・例外管理表

---

## 10. 作業⑧ 品質評価・報告

### 定期報告内容

* 品質KPI達成状況
* 不具合件数推移
* 標準準拠状況

👉 **自治体決裁向けに「一目で分かる」資料が必須**

---

## 11. 作業⑨ 監査・検査対応

* 監査観点一覧の事前整理
* 成果物・証跡の即時提示可能化
* 口頭説明用シナリオ整備

---

## 12. PL/PM支援としての実務ポイント

* 「品質＝SIer任せ」にしない
* 文書品質・説明品質を重視
* 標準準拠を**設計思想レベル**で統制
* 将来ベンダー交代を見据える

---

### まとめ（重要）

自治体システム標準準拠における品質管理計画とは、

> **「作ったものが正しい」ではなく
> 「なぜ正しいかをいつでも説明できる状態を作る計画」**

です。

# ◆ 品質管理計画書（自治体決裁用）

---

## 1. 文書概要

### 1.1 文書目的

本品質管理計画書は、自治体システム標準準拠（デジタル庁標準仕様書・ガバメントクラウド利用前提）に基づき実施される本プロジェクトにおいて、各工程で確保すべき品質方針、品質基準、品質管理方法および管理体制を明確化し、成果物の妥当性・完全性・標準準拠性を確実に担保することを目的とする。

### 1.2 適用範囲

本計画書は、以下の工程および成果物に適用する。

* 要件定義工程
* 設計工程（基本設計・詳細設計）
* 開発・設定工程
* テスト工程
* 移行・本番切替工程

### 1.3 関連文書

* デジタル庁 標準仕様書（対象業務システム）
* プロジェクト計画書
* 要件定義書
* 試験計画書
* 移行計画書

---

## 2. 品質方針および品質目標

### 2.1 品質方針

本プロジェクトでは、デジタル庁が定める標準仕様書への完全準拠を最優先とし、自治体業務の継続性および住民サービスへの影響を最小化することを品質方針とする。
また、将来の制度改正、他団体展開、ベンダー交代を考慮し、説明可能性および再現性の高い成果物を作成する。

### 2.2 品質目標（KPI）

| 区分   | 品質目標           |
| ---- | -------------- |
| 標準準拠 | 標準仕様適合率 100%   |
| レビュー | 指摘事項是正完了率 100% |
| 不具合  | 本番障害 0件        |
| 文書品質 | 承認未取得成果物 0件    |

---

## 3. 品質管理体制

### 3.1 体制概要

品質管理は、発注者（自治体）、受注者（SIer）、PL/PM支援者が役割分担のもと実施する。

### 3.2 役割・責任

| 役割       | 主な責任              |
| -------- | ----------------- |
| 発注者（自治体） | 成果物承認、品質判断、最終責任   |
| PL/PM支援  | 品質統制、第三者レビュー、是正指示 |
| SIer PM  | 工程管理、成果物品質確保      |
| QA担当     | レビュー運営、不具合管理      |

---

## 4. 管理対象成果物および品質基準

### 4.1 管理対象成果物

| 工程   | 成果物               |
| ---- | ----------------- |
| 要件定義 | 要件定義書、Fit & Gap一覧 |
| 設計   | 基本設計書、詳細設計書       |
| 開発   | ソースコード、設定定義書      |
| テスト  | 試験計画書、試験結果報告書     |
| 移行   | 移行計画書、移行検証結果      |

### 4.2 品質基準

* 完全性：必須項目の記載漏れがないこと
* 妥当性：業務要件および法令要件と整合していること
* 一貫性：他成果物と矛盾がないこと
* 標準準拠性：標準仕様書への適合が確認できること
* 証跡性：判断根拠が文書で明示されていること

---

## 5. 工程別品質管理方法

### 5.1 要件定義工程

* 標準仕様項番単位でのトレーサビリティ管理
* Fit & Gap妥当性レビューの実施
* 非機能要件（性能・可用性・セキュリティ）の重点確認

### 5.2 設計工程

* 要件トレーサビリティ確認
* 標準機能優先設計の遵守確認
* 個別カスタマイズ抑制チェック

### 5.3 開発工程

* コーディング規約遵守確認

### 5.4 テスト工程

### 6.1 レビュー種別

| ------- | ------- |

| 第三者レビュー | PL/PM支援 |

### 6.2 主なレビュー観点

* 標準仕様準拠性

* 将来制度改正耐性
* 他団体展開可能性

---

## 7. 不具合管理および是正管理

* 不具合は重大度（Critical/High/Medium/Low）で管理する
* 原因分析および再発防止策を必ず記録する
* 是正完了は再発防止策の確認をもって完了とする

---

## 8. 標準仕様適合性管理

* 標準仕様項番ごとの対応表を作成する
* 未対応または代替対応がある場合は理由を明記する
* 標準仕様改訂時は影響評価を実施する

---

## 9. 品質評価および報告

* 定例会にて品質状況を報告する
* 品質KPI、不具合件数、標準準拠状況を可視化する
* 重大な品質リスクは速やかに共有する

---

## 10. 監査・検査対応

* 監査観点を事前整理し、成果物・証跡を即時提示可能とする
* 監査対応時の説明責任はPL/PM支援およびSIerが補佐する

---

この品質管理計画書の内容は、

* ✅ 自治体内部決裁に耐える構成
* ✅ デジタル庁標準仕様・標準準拠監査を意識
* ✅ 「品質＝説明責任」を前提とした記述
* ✅ PL/PM支援が前面に出過ぎないが、統制が効く設計

を意図しています。

---

### 実務での使い方（重要）

* **そのまま決裁資料に添付可能**
* 個別案件では以下だけ差し替えれば十分です

  * 対象業務（例：住民記録／税／福祉 等）
  * 適用標準仕様名
  * 体制（組織名・役割名）
