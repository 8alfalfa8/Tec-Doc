# ◆ 品質管理（QC）

以下では、**ITシステム開発の全工程（企画〜運用）における品質管理**（**QC: Quality Control**）を、
**各工程ごとに**「**管理目的／品質方針／管理内容／方法・手法／注意事項／成果物**」の観点で体系的に整理します。
※大規模基幹系・クラウド（AWS）・アジャイル／ウォーターフォール双方に適用できる汎用構造です。

---

## 1. 品質管理（QC）の全体像

### 1.1 QCの定義

* **品質管理（QC）**：
  定義された品質基準を満たす成果物を**計画的・継続的に作り込むための管理活動**
* 単なる「テスト」ではなく
  👉 **全工程における予防活動＋検証活動**

### 1.2 品質の主要観点（ISO/IEC 25010）

| 観点     | 内容          |
| ------ | ----------- |
| 機能適合性  | 要求通りに動く     |
| 性能効率   | 応答時間、スループット |
| 信頼性    | 障害耐性、可用性    |
| 使用性    | UX、操作性      |
| 保守性    | 変更容易性       |
| セキュリティ | 機密性・完全性     |
| 互換性    | 他システム連携     |
| 移植性    | 環境差異対応      |

## 2. 全工程QCマップ（俯瞰）

| 工程      | 主な品質活動       |
| ------- | ------------ |
| 企画・構想   | 要求の妥当性・実現性   |
| 要件定義    | 要件の完全性・一貫性   |
| 基本設計    | 構造の正しさ・非機能   |
| 詳細設計    | 実装可能性・テスト容易性 |
| 実装      | コード品質        |
| テスト     | 要求適合性        |
| 移行・リリース | 安定性・完全性      |
| 運用・保守   | 継続的品質改善      |

## 3. 工程別 品質管理（QC）詳細

---

### 3.1 企画・構想工程

#### 管理目的

* **作るべきものが正しいか**（Why / What）

#### 品質方針

* 「作らないリスク」より「作る価値」を明確に
* ビジネス妥当性を最優先

#### 管理内容

* 課題定義の妥当性
* ROI / TCOの妥当性
* スコープ定義の明確性

#### 方法・手法

* ビジネス要求レビュー
* ステークホルダー合意形成
* PoC / フィージビリティ検証

#### 注意事項

* 要求が抽象的すぎる
* 技術先行にならない

#### 成果物

* システム構想書
* ビジネス要件定義書
* 投資対効果試算

---

### 3.2 要件定義工程

#### 管理目的

* **要求の漏れ・曖昧さ・矛盾を排除**

#### 品質方針

* 「後工程で直さない」
* **曖昧表現ゼロ**

#### 管理内容

* 機能要件の完全性
* 非機能要件（性能・可用性・セキュリティ）
* 業務ルール整合性

#### 方法・手法

* 要件レビュー（第三者含む）
* トレーサビリティマトリクス
* ユースケース / 業務フロー

#### 注意事項

* 「想定」「原則」「可能であれば」はNG
* 非機能要件の後回し

#### 成果物

* 要件定義書
* 非機能要件一覧
* 要件トレーサビリティ表（RTM）

---

### 3.3 基本設計（外部設計）

#### 管理目的

* **要件を満たす構造か**

#### 品質方針

* 拡張性・保守性重視
* 標準化・再利用優先

#### 管理内容

* アーキテクチャ妥当性
* 外部IF設計
* 画面・帳票設計

#### 方法・手法

* アーキテクチャレビュー
* 非機能レビュー
* 設計標準チェック

#### 注意事項

* 技術者都合設計
* 非機能の設計漏れ

#### 成果物

* 基本設計書
* システム構成図
* IF仕様書

---

### 3.4 詳細設計（内部設計）

#### 管理目的

* **実装ミスが起きない設計**

#### 品質方針

* 可読性・テスト容易性

#### 管理内容

* 処理フロー正確性
* データ設計妥当性
* 例外処理・エラー設計

#### 方法・手法

* 設計レビュー
* 疑似コード
* テスト観点洗い出し

#### 注意事項

* 実装者依存設計
* ブラックボックス化

#### 成果物

* 詳細設計書
* ER図
* 処理フロー図

---

### 3.5 実装（製造）

#### 管理目的

* **バグを作り込まない**

#### 品質方針

* 自動化・標準遵守

#### 管理内容

* コーディング規約遵守
* 静的品質
* 単体テスト品質

#### 方法・手法

* コードレビュー
* 静的解析（SonarQube等）
* 単体テスト（カバレッジ）

#### 注意事項

* レビュー形骸化
* テスト不足

#### 成果物

* ソースコード
* 単体テスト仕様書・結果

---

### 3.6 テスト工程（結合・総合・受入）

#### 管理目的

* **要件を満たしているか**

#### 品質方針

* 要件ベーステスト
* 再現性重視

#### 管理内容

* テスト網羅性
* 不具合管理
* 再発防止

#### 方法・手法

* テスト設計レビュー
* 欠陥管理表
* 自動テスト

#### 注意事項

* テストケース不足
* 直しっぱなし

#### 成果物

* テスト仕様書
* テスト結果報告書
* 障害管理表

---

### 3.7 移行・リリース

#### 管理目的

* **安全に本番へ**

#### 品質方針

* 失敗前提設計

#### 管理内容

* データ移行正確性
* リリース手順妥当性
* ロールバック可否

#### 方法・手法

* リハーサル
* チェックリスト
* Go/No-Go判定

#### 注意事項

* 手順属人化
* 想定外シナリオ不足

#### 成果物

* 移行計画書
* リリース手順書

---

### 3.8 運用・保守

#### 管理目的

* **品質の維持・改善**

#### 品質方針

* 継続的改善（PDCA）

#### 管理内容

* 障害傾向分析
* SLA/KPI管理
* 変更管理

#### 方法・手法

* インシデント管理
* 問題管理
* 定期レビュー

#### 注意事項

* 応急対応のみ
* 根本原因分析不足

#### 成果物

* 運用報告書
* 改善計画

---

## 4. QC成功のための重要ポイント

| ポイント     | 内容          |
| -------- | ----------- |
| 早期品質作り込み | 上流で8割決まる    |
| 第三者視点    | 独立QA        |
| 定量指標     | バグ密度、再発率    |
| 自動化      | CI/CD＋自動テスト |

---

## 5. まとめ（要点）

* QCは**全工程活動**
* 上流品質＝下流コスト削減
* 「レビュー × トレーサビリティ × 定量指標」が鍵
* テストはQCの一部にすぎない
