# ◆ アジャイル品質管理における実務テンプレート

以下では、**そのままプロジェクトで使える実務テンプレート**として
① **DoR（Definition of Ready）チェックリスト**
② **DoD（Definition of Done）チェックリスト**
③ **アジャイル品質KPI定義書（標準版）**
を **Scrum／SAFe／DevOps共通**で整理します。

---

## ① DoR（Definition of Ready）チェックリスト

※「**スプリントに入れてよいか**」を判断する品質ゲート

### 1. DoRの位置づけ

* 目的：**未成熟な要求をスプリントに持ち込まない**
* 効果：再作業・手戻り・品質低下を防止
* 責任：**PO主導＋チーム合意**

---

### 2. DoRチェックリスト（標準）

#### A. ビジネス・要求品質

| No | チェック項目              | 確認 |
| -- | ------------------- | -- |
| R1 | ユーザーストーリー形式で記述されている | ☐  |
| R2 | 価値（Why）が明確          | ☐  |
| R3 | スコープが明確で大きすぎない      | ☐  |
| R4 | 曖昧語（適切に／可能な限り 等）がない | ☐  |
| R5 | 優先度が決定済み            | ☐  |

---

#### B. 受入条件（AC）・テスト観点

| No | チェック項目                    | 確認 |
| -- | ------------------------- | -- |
| R6 | 受入条件（AC）が明記されている          | ☐  |
| R7 | 正常系・異常系が含まれている            | ☐  |
| R8 | テスト可能な表現になっている            | ☐  |
| R9 | 非機能要件（性能/セキュリティ等）が考慮されている | ☐  |

---

#### C. 技術・実装準備

| No  | チェック項目            | 確認 |
| --- | ----------------- | -- |
| R10 | 技術的実現性が確認済み       | ☐  |
| R11 | 依存関係が明確           | ☐  |
| R12 | 技術スパイクが必要な場合は実施済み | ☐  |
| R13 | 見積がチームで合意されている    | ☐  |

---

#### D. 運用・リリース観点

| No  | チェック項目                | 確認 |
| --- | --------------------- | -- |
| R14 | 影響範囲（他機能・データ）が把握されている | ☐  |
| R15 | ログ・監視・アラート観点が考慮されている  | ☐  |

---

#### ✅ DoR合格基準

* **全項目が ☐＝OK**
* 未達項目は**バックログへ戻す**

---

## ② DoD（Definition of Done）チェックリスト

※「**完了と見なしてよいか**」の品質定義

### 1. DoDの位置づけ

* 目的：**「動く」≠「完了」を防ぐ**
* 効果：品質のばらつき排除
* 責任：**チーム全員**

---

### 2. DoDチェックリスト（標準）

#### A. 実装品質

| No | チェック項目          | 確認 |
| -- | --------------- | -- |
| D1 | コーディング規約遵守      | ☐  |
| D2 | 可読性・保守性が確保されている | ☐  |
| D3 | 技術的負債を増やしていない   | ☐  |

---

#### B. テスト品質

| No | チェック項目      | 確認 |
| -- | ----------- | -- |
| D4 | 単体テスト作成・成功  | ☐  |
| D5 | カバレッジ基準達成   | ☐  |
| D6 | 自動テストがCIで実行 | ☐  |
| D7 | リグレッション確認済み | ☐  |

---

#### C. レビュー・品質確認

| No  | チェック項目      | 確認 |
| --- | ----------- | -- |
| D8  | コードレビュー完了   | ☐  |
| D9  | 指摘事項は全て解消   | ☐  |
| D10 | 静的解析警告が許容範囲 | ☐  |

---

#### D. 要求・受入

| No  | チェック項目         | 確認 |
| --- | -------------- | -- |
| D11 | 受入条件（AC）全達成    | ☐  |
| D12 | POまたは代理による受入確認 | ☐  |

---

#### E. 運用・リリース

| No  | チェック項目      | 確認 |
| --- | ----------- | -- |
| D13 | ログ・監視設定済み   | ☐  |
| D14 | リリース影響説明可能  | ☐  |
| D15 | ドキュメント最小限更新 | ☐  |

---

#### ✅ DoD合格基準

* **全項目達成で「Done」**
* 一部免除は**チーム合意＋記録必須**

## ③ アジャイル品質KPI定義書（標準）

### 1. KPI設計方針

* **少数・行動に直結**
* **スプリント単位で改善可能**
* **評価目的に使わない**

---

### 2. 必須品質KPI（最小セット）

#### KPI-01：欠陥流出率（最重要）

| 項目   | 定義                |
| ---- | ----------------- |
| 目的   | 上流品質評価            |
| 定義   | 後工程で検出された欠陥 / 全欠陥 |
| 単位   | %                 |
| 目標   | 20%以下             |
| 改善施策 | DoR強化、レビュー強化      |

---

#### KPI-02：再作業率

| 項目   | 定義          |
| ---- | ----------- |
| 目的   | 作り込み品質      |
| 定義   | 再作業工数 / 全工数 |
| 単位   | %           |
| 目標   | 10〜15%以下    |
| 改善施策 | AC明確化、TDD   |

---

#### KPI-03：自動テスト率

| 項目   | 定義                |
| ---- | ----------------- |
| 目的   | 予防力               |
| 定義   | 自動テスト実行ケース / 全テスト |
| 単位   | %                 |
| 目標   | 70%以上             |
| 改善施策 | CI整備、テスト投資        |

---

#### KPI-04：変更失敗率（DevOps）

| 項目   | 定義            |
| ---- | ------------- |
| 目的   | リリース品質        |
| 定義   | 障害を伴う変更 / 全変更 |
| 単位   | %             |
| 目標   | 15%以下         |
| 改善施策 | DoD強化、段階リリース  |

---

#### KPI-05：MTTR

| 項目   | 定義         |
| ---- | ---------- |
| 目的   | 回復力        |
| 定義   | 障害復旧平均時間   |
| 単位   | 時間         |
| 目標   | SLA以内      |
| 改善施策 | 監視・Runbook |

---

### 3. 補助KPI（任意）

| KPI         | 目的   |
| ----------- | ---- |
| DoR達成率      | 要求品質 |
| DoD未達理由Top3 | 改善焦点 |
| COQ失敗比率     | 投資判断 |
| バグ再発率       | 根本対策 |

---

## 4. 実運用ルール（重要）

* KPIは**トレンドを見る**
* 数値悪化＝**改善テーマ**
* 人事評価と切り離す
* レトロで必ず議論

---

## 5. まとめ

* **DoR＝入れてよいか**
* **DoD＝終わってよいか**
* KPI＝**改善を回すための計器**
* この3点が揃うとアジャイルQCは安定する
