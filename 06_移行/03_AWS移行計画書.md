## 1. 目的・背景
[🔙 目次に戻る](#index)


本計画書は、既存オンプレミス基幹業務システムを AWS 上へ段階的に移行し、
可用性向上、運用負荷削減、将来的なDX対応を実現することを目的とする。

背景：

* オンプレ機器老朽化（保守期限切れ）
* 障害時の復旧に時間を要している
* 環境増設に時間とコストがかかる

---

## 2. 移行対象システム概要
[🔙 目次に戻る](#index)


### 2.1 現行（As-Is）構成
[🔙 目次に戻る](#index)


* Web/APサーバ：物理サーバ 8台（Tomcat）
* バッチサーバ：物理サーバ 2台
* DB：Oracle RAC（2ノード）
* 外部IF：SFTP連携（5社）、内部基幹IF（3系）

### 2.2 移行後（To-Be）AWS構成
[🔙 目次に戻る](#index)


* Web/AP（業務A）：ECS（Fargate）
* Web/AP（業務B）：EKS（マイクロサービス）
* バッチ：EC2（AutoScalingなし）
* DB：Amazon RDS for Oracle（Multi-AZ）
* 共通：ALB / IAM / CloudWatch / CloudTrail

---

## 3. 移行スコープ
[🔙 目次に戻る](#index)


### 3.1 移行対象
[🔙 目次に戻る](#index)


* 業務アプリA（Web）
* 業務アプリB（API）
* バッチ処理
* 業務データ（過去5年分）

### 3.2 移行対象外
[🔙 目次に戻る](#index)


* 帳票出力基盤（別PJで刷新）
* 過去ログ（S3へアーカイブのみ）

---

## 4. AWS移行戦略（6R）
[🔙 目次に戻る](#index)


| システム | 移行方式        | 理由          |
| ---- | ----------- | ----------- |
| 業務A  | Replatform  | コンテナ化で運用削減  |
| 業務B  | Rearchitect | 将来マイクロサービス化 |
| バッチ  | Rehost      | 改修コスト最小     |
| DB   | Replatform  | RDS運用負荷削減   |

### ※ この例の実務的ポイント
[🔙 目次に戻る](#index)


#### ① 実案件で「そのまま通る」構成
[🔙 目次に戻る](#index)


* **6Rでの移行判断理由を明文化**
* **Wave分割（検証→業務→基幹）を明確化**
* **ECS / EKS の使い分け理由が説明可能**
* **RDS（Multi-AZ）前提で監査・可用性を担保**

#### ② PM／ITコンサル視点で評価されやすい点
[🔙 目次に戻る](#index)


* 切替・ロールバック手順を**計画書段階で記載**
* データ移行を **DMS前提＋差分同期**まで明示
* RACIで「責任の所在」が曖昧にならない
* コストも **稟議で使える粒度**

#### ③ よくあるNGを回避
[🔙 目次に戻る](#index)


* 「全部EKS」になっていない
* 切替当日の手順が抽象論で終わっていない
* 運用引継ぎが計画に含まれている
