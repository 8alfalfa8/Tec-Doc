# ◆ SPA（Single Page Application）における認証・認可設計

以下では、**SPA（Single Page Application）における認証・認可設計**について、
**概念整理 → 脅威モデル → 方式比較 → 推奨アーキテクチャ → 詳細設計（フロー／実装） → 非機能・運用 → 設計成果物 → 失敗例**
の順で、**金融・公共・大規模業務システムでそのまま設計書に使えるレベル**まで掘り下げて説明します。

---

## 1. 認証・認可の前提整理（最重要）
[🔙 目次に戻る](#index)


### 1.1 認証（Authentication）
[🔙 目次に戻る](#index)


> **「あなたは誰か」**

* ログイン
* セッション確立
* Token検証

### 1.2 認可（Authorization）
[🔙 目次に戻る](#index)


> **「あなたは何をしてよいか」**

* 画面表示可否
* API実行可否
* データ参照範囲

👉 **SPAではこの2つを明確に分離して設計する**

---

## 2. SPA特有の脅威モデル
[🔙 目次に戻る](#index)


### 2.1 SPAが抱えるリスク
[🔙 目次に戻る](#index)


| 脅威      | 説明           |
| ------- | ------------ |
| XSS     | JSからToken窃取  |
| CSRF    | Cookie悪用     |
| Token漏洩 | LocalStorage |
| 権限偽装    | クライアント改変     |
| API直叩き  | BFFなし        |

---

### 2.2 結論（業務系の原則）
[🔙 目次に戻る](#index)


> **「認証情報をJavaScriptに触らせない」**

---

## 3. 認証方式の比較
[🔙 目次に戻る](#index)


| 方式                    | 評価 | コメント   |
| --------------------- | -- | ------ |
| Session + Cookie      | ○  | スケール注意 |
| JWT + LocalStorage    | ❌  | XSSで即死 |
| JWT + HttpOnly Cookie | ◎  | 推奨     |
| OAuth2 / OIDC         | ◎  | IdP連携可 |

👉 **金融・公共では OIDC + Cookie が標準**

---

## 4. 推奨アーキテクチャ（結論）
[🔙 目次に戻る](#index)


```
[ Browser (SPA) ]
      |
      | HTTPS (Cookie)
      v
[    BFF    ]
      |
      v
[ Backend / IdP ]
```

#### 役割分担
[🔙 目次に戻る](#index)


| 層       | 役割      |
| ------- | ------- |
| SPA     | UI制御のみ  |
| BFF     | 認証・認可判断 |
| Backend | 業務ロジック  |
| IdP     | ユーザー管理  |

---

## 5. 認証設計（詳細）
[🔙 目次に戻る](#index)


### 5.1 ログインフロー（OIDC例）
[🔙 目次に戻る](#index)


```
① SPA → /login
② BFF → IdP リダイレクト
③ IdP 認証
④ 認可コード
⑤ BFF Token取得
⑥ HttpOnly Cookie保存
```

👉 **TokenはBFFでのみ保持**

---

### 5.2 Cookie設計（重要）
[🔙 目次に戻る](#index)


| 属性       | 設定           |
| -------- | ------------ |
| HttpOnly | true         |
| Secure   | true         |
| SameSite | Lax / Strict |
| Domain   | 最小範囲         |
| Path     | /            |

# ◆ SPA × BFF × マルチテナント環境における認可設計

以下では、**SPA × BFF × マルチテナント環境における認可設計**について、
**概念 → テナントモデル → 認可レイヤ → 設計原則 → データ／API設計 → フロー → 実装指針 → 非機能 → 成果物 → 失敗例**
の順で、**金融・公共・SaaSでそのまま設計書に落とせるレベル**まで詳しく説明します。

---

## 1. マルチテナント認可とは（前提）
[🔙 目次に戻る](#index)


### 1.1 定義
[🔙 目次に戻る](#index)


**1つのアプリケーション基盤を複数の「テナント（組織・顧客）」が安全に共有し、
他テナントのデータ・操作を一切参照・操作できないようにする認可設計**

👉 認証より **認可の難易度が高い**

---

## 2. マルチテナントで守るべき境界
[🔙 目次に戻る](#index)


| 境界     | 説明       |
| ------ | -------- |
| テナント境界 | 他社データ遮断  |
| ロール境界  | 管理者 / 一般 |
| データ境界  | 行・列レベル   |
| 機能境界   | 契約プラン    |

👉 **境界が4層あるのが業務系の現実**

---

## 3. テナントモデルの種類（設計起点）
[🔙 目次に戻る](#index)


### 3.1 テナント分離方式（重要）
[🔙 目次に戻る](#index)


| 方式     | 概要         | 評価      |
| ------ | ---------- | ------- |
| DB分離   | テナント毎DB    | ◎（金融）   |
| スキーマ分離 | 1DB多Schema | ○       |
| 行分離    | tenant_id  | ◎（SaaS） |

👉 認可設計は **分離方式に強く依存**

---

### 3.2 推奨（一般SaaS）
[🔙 目次に戻る](#index)


* **行分離 + tenant_id**
* 物理分離は将来オプション

---
