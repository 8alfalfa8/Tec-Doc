# SOC（Security Operations Center）構築における具体的作業

以下では、**SOC（Security Operation Center）構築**について、
**実務でそのまま使えるレベル**で、
**作業タスク／入力情報／作業内容／方法（手順）／成果物／注意事項**の観点から体系的に詳述します。
（※CSIRTは組織・判断・対外対応、SOCは**24/7監視・検知・分析・一次対応**が主眼です）

---

## 1. SOC構築の全体像

### 1.1 SOCの目的

* セキュリティイベントの**常時監視**
* インシデントの**早期検知・影響最小化**
* CSIRTへの**正確・迅速なエスカレーション**
* 監査・法令対応（ログ保全・証跡）

---

### 1.2 SOCの主要機能

| 領域   | 内容            |
| ---- | ------------- |
| 監視   | ログ・アラートの常時監視  |
| 検知   | 攻撃兆候・異常行動の検知  |
| 分析   | 影響範囲・深刻度の評価   |
| 初動対応 | 一次切り分け・封じ込め   |
| 報告   | CSIRT/経営層への報告 |
| 改善   | ルール・運用改善      |

# AWS特化 SOC構築における具体的作業

以下は、**AWS環境に特化したSOC（Security Operation Center）構築例**を、
**実案件レベル**で「タスク／入力情報／作業内容／方法／成果物／注意事項」まで落として整理したものです。
（※ 先にご説明したSOC一般論を **AWSサービスに完全対応** させています）

---

## AWS特化 SOC構築 全体像

### 目的（AWS前提）

* **CloudTrail / GuardDuty / Security Hub / VPC Flow Logs** 等を中心に
  AWSネイティブ＋SIEMで **統合監視**
* マルチアカウント・マルチリージョン前提
* **CSIRTと即連携できる実運用SOC**

---

## 1. AWS特化SOC アーキテクチャ全体像

```
[AWS Accounts (複数)]
   ├ CloudTrail
   ├ VPC Flow Logs
   ├ ELB / ALB Logs
   ├ CloudWatch Logs
   ├ GuardDuty
   ├ Security Hub
   ├ AWS Config
   └ EDR (CrowdStrike等)
        ↓
[Log Archive Account]
   ├ S3 (ログ集約・WORM)
   ├ KMS
        ↓
[SIEM Account]
   ├ Amazon OpenSearch / Splunk / Sentinel
   ├ SOAR (Step Functions / Lambda)
        ↓
[SOCオペレーター]
        ↓
[CSIRT / 経営層]
```

# SOC × CSIRT 連携Runbook（AWS特化）

以下は、**実運用でそのまま使えるレベル**の
**「SOC × CSIRT 連携Runbook（AWS前提）」**です。
**誰が・いつ・何を判断し・どこまで対応するか**を明確にし、
**夜間・誤検知・重大事故**でも破綻しない構成にしています。

---

## 1. 本Runbookの位置づけ

### 目的

* SOC（監視・分析）とCSIRT（判断・統括・対外対応）の**責任分離**
* 判断遅延・責任不明確による被害拡大を防止
* AWSインシデントの**標準対応化**

### 適用範囲

* AWSマルチアカウント環境
* GuardDuty / Security Hub / SIEM検知
* 内部不正・外部攻撃・設定不備

---

## 2. SOCとCSIRTの役割分担（原則）

### 基本原則

> **SOCは「事実確認」まで、CSIRTは「意思決定」から**

| 項目       | SOC | CSIRT |
| -------- | --- | ----- |
| 常時監視     | ○   | −     |
| 初動分析     | ○   | −     |
| 影響範囲推定   | ○   | 補足    |
| 封じ込め判断   | −   | ○     |
| システム停止判断 | −   | ○     |
| 対外連絡     | −   | ○     |
| 経営報告     | −   | ○     |

---

## 3. インシデントレベル定義（共通言語）

### レベル分類（AWS向け）

| レベル | 定義     | 主な例      | 主体          |
| --- | ------ | -------- | ----------- |
| Lv1 | 軽微・誤検知 | 正常API誤検知 | SOC         |
| Lv2 | 要調査    | 異常ログイン試行 | SOC→CSIRT通知 |
| Lv3 | 侵害疑い   | IAM侵害兆候  | CSIRT主導     |
| Lv4 | 侵害確定   | Root侵害   | CSIRT緊急対応   |

---

## 4. 全体フロー（標準）

```
① 検知（SOC）
   ↓
② 初動分析（SOC）
   ↓
③ レベル判定（SOC仮判定）
   ↓
④ CSIRTエスカレーション
   ↓
⑤ 封じ込め判断（CSIRT）
   ↓
⑥ 対応実施（SOC/Infra）
   ↓
⑦ 報告・記録（CSIRT）
```

---

## 5. フェーズ別 Runbook 詳細

---

### フェーズ① 検知（SOC）

#### トリガー

* GuardDuty Finding
* SIEM相関アラート
* AWS Config違反

#### SOC作業

* アラート受付
* チケット発行（自動）

#### 成果物

* インシデント管理票（起票）

#### 注意事項

* **即判断しない**
* 全件ログ保全開始

---

### フェーズ② 初動分析（SOC）

#### SOC分析項目（AWS共通）

* 該当アカウント／リージョン
* 該当IAMユーザー／Role
* 発生時刻（UTC/JST）
* 関連CloudTrailイベント
* 過去同様事象有無

#### 利用ツール

* AWS Console
* SIEM
* Athena（CloudTrail）

#### 成果物

* 初動分析メモ（定型）

#### 注意事項

* **対応操作は行わない**
* 証拠改変禁止

---

### フェーズ③ レベル仮判定（SOC）

#### 判定基準

| 観点   | チェック     |
| ---- | -------- |
| 権限   | 管理者権限か   |
| 持続性  | 永続IAM変更  |
| 外部通信 | 不審IP     |
| 横展開  | 他アカウント波及 |

#### SOC判断

* Lv1〜Lv4 仮分類
* Lv2以上 → CSIRT通知

#### 成果物

* エスカレーション通知

---

### フェーズ④ CSIRTエスカレーション

#### 通知内容（必須）

* インシデントID
* 仮レベル
* 事実（推測禁止）
* 影響範囲（暫定）
* 推奨対応案（参考）

#### 通知手段

* 平日：Teams / Slack
* 夜間：電話＋SMS

#### 注意事項

* **推測・感情禁止**
* 「事実のみ」

---

### フェーズ⑤ 封じ込め判断（CSIRT）

#### CSIRT判断項目

* 業務影響
* 法令影響（個情法等）
* 停止可否
* 自動対応可否

# AWS SOC KPI / SLA 定義ガイド

以下は、**AWS環境に特化したSOC KPI / SLA定義**を、

**「測れる」「ごまかせない」「改善に直結する」**ことを重視しています。

---

## 1. KPI / SLA の基本思想（重要）

### 原則

1. **AWSサービス起点で測定できる**
2. **SOCでコントロール可能**
3. **CSIRT判断と切り分ける**
4. **改善アクションに直結**

> ❌ NG：
> 「セキュリティ事故ゼロ」
> 「重大インシデント未発生」

---

## 2. KPI / SLA 全体構造

```
【Input】   ログ・アラート
【Process】検知 → 分析 → 通知
【Output】  対応・報告
【Outcome】被害抑止（参考指標）
```

# AWS SOC／CSIRT 監査・ISMS向け説明資料

## 1. 目的と位置づけ

本資料は、当社AWS環境におけるSOC（Security Operation Center）およびCSIRT（Computer Security Incident Response Team）の体制・運用について、**監査人・ISMS審査員向けに説明すること**を目的とする。

* ISMS（ISO/IEC 27001）要求事項への適合性説明
* 内部統制・監査対応（J-SOX、内部監査）
* SOC／CSIRTの役割分担・責任範囲の明確化

---

## 2. SOC／CSIRTの全体像

### 2.1 基本コンセプト

* **SOC**：24/7監視・検知・初動分析を担う実行組織
* **CSIRT**：判断・統括・対外対応を担う意思決定組織

> 原則：SOCは「事実確認まで」、CSIRTは「意思決定から」

### 2.2 組織構成（概要）

* 経営層（最終責任）
* CSIRT（CISO／情報セキュリティ責任者）
* SOC（運用チーム、外部委託含む）

---

## 3. AWS環境におけるSOC構成

### 3.1 対象範囲

* AWS Organizations配下の全アカウント
* 全リージョン
* 本番・検証環境

### 3.2 利用AWSサービス（監査観点）

* CloudTrail：操作証跡の取得・保全
* GuardDuty：脅威検知
* Security Hub：統合可視化
* VPC Flow Logs：通信監視
* S3（Object Lock）：ログ改ざん防止

---

## 4. ISMS要求事項との対応関係

| A.8.16 監視活動         | 24/7 SOC運用           |

## 5. インシデント管理プロセス（監査説明用）

### 5.1 標準フロー

1. 検知（SOC）
2. 初動分析（SOC）
3. レベル仮判定（SOC）
4. エスカレーション（CSIRT）
5. 対応判断・指示（CSIRT）
6. 対応実施（SOC／インフラ）
7. 報告・是正・再発防止

### 5.2 証跡管理

* 全操作はCloudTrailに記録
* SOC対応履歴はチケット管理システムで保存
* 保存期間：○年（社内規程準拠）

## 6. KPI／SLAによる統制

### 6.1 SLA（運用品質の保証）

* 検知遅延：5分以内
* 初動分析完了：30分以内
* 夜間監視カバレッジ：100%

### 6.2 KPI（改善指標）

* 誤検知率
* MTTA（平均分析時間）
* エスカレーション適正率

> KPIは月次でレビューし、改善計画を策定

## 8. 監査・審査時によくある質問（想定QA）

**Q. SOCが勝手にシステムを止めることはないか？**
A. 封じ込め・停止判断はCSIRTのみが実施し、SOCは指示に基づく操作のみを行う。

**Q. ログ改ざん防止は？**
A. CloudTrailログは専用S3に集約し、Object LockでWORM化している。

**Q. 夜間・休日の対応は？**
A. 24/7監視体制とCSIRTオンコール体制を確立している。
