# ◆ オンプレミスシステムからAWS移行における運用設計

以下では、**オンプレミスシステムをAWSへ移行する場合に特有となる**「**運用設計作業**」を、
**移行フェーズを前提にしたタスク構造**で、
**現場でそのまま運用設計書・移行計画書・RFPに使える粒度**まで落として詳しく説明します。

> ポイント
>
> * 通常の運用設計 ＋ 「**移行期間特有の運用**」が必須
> * 旧環境（オンプレ）と新環境（AWS）が**同時に存在する前提**で設計する

---

## 1. 全体像：オンプレ → AWS移行における運用設計の位置づけ

### 1.1 フェーズ別の運用設計範囲

| フェーズ | 運用設計の主眼         |
| ---- | --------------- |
| 移行前  | 現行運用の可視化・棚卸     |
| 移行設計 | AWS前提の新運用設計     |
| 移行期間 | **二重運用・切替運用設計** |
| 移行後  | AWSネイティブ運用      |

---

### 1.2 運用設計タスク全体構造

```
① 現行（オンプレ）運用調査・棚卸
② 運用差分分析（オンプレ vs AWS）
③ 移行方針・運用設計方針策定
④ AWS前提の運用体制設計
⑤ 二重運用・移行期間運用設計
⑥ AWS基盤運用設計
⑦ 監視・ログ・証跡運用設計
⑧ 障害・インシデント運用設計
⑨ 変更・リリース運用設計
⑩ セキュリティ運用設計
⑪ バックアップ・DR運用設計
⑫ 運用手順書整備・教育
⑬ 運用移管・安定化対応
```

---

## 3. 移行案件で必ず失敗しやすいポイント

❌ オンプレ運用をそのままAWSに移植<br>
❌ 移行期間の運用を軽視<br>
❌ 「誰が止めるか」が曖昧<br>
❌ 自動化を後回し

---

## 4. まとめ（重要）

> **オンプレ → AWS移行の運用設計は
> 「移行作業」ではなく「運用変革」**
