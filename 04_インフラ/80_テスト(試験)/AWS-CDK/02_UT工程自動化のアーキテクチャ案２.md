# ◆ UT工程自動化のアーキテクチャ案２

AWS CDKにおける単体テストの「完全自動化」と「カバレッジ100%」を実現するための、**アーキテクチャ提案書（案2：高度自動化モデル）** を整理します。

本案の核心は、既存ツールで補えない **「ソースコードの論理構造（if文など）とテストコードの紐付け」** を、軽量な自作Parser（解析スクリプト）で解決する点にあります。

---
## 1. 目的と導入効果
*   **カバレッジ100%の強制**: 手動テストによる漏れを排除し、全分岐の検証を自動化。
*   **ドキュメント・コードの完全同期**: ソースコードから仕様書とテストを逆生成し、乖離をゼロにする。
*   **エビデンス収集の無人化**: GitLab CI上で全ての証跡（仕様・コード・結果・ログ）を自動パッケージングする。

---

## 2. 全体アーキテクチャ・フロー
GitLab CIパイプラインを軸とした、開発サイクル全体の構成です。

1.  **静的解析 (Lint/Check)**: `cdk-nag` によるセキュリティチェックと環境変数確認。
2.  **解析・生成 (Spec & Test Gen)**: **自作Parser**がCDKコードを解析し、仕様書とテストコードを自動出力。
3.  **実行 (Unit Test)**: Jestが起動。`TestEnvironmentManager` によりテストごとに環境を完全リセットして実行。
4.  **エビデンス集計 (Archiving)**: 生成物と実行結果を一つのエビデンスパッケージとして保存。

---

## 3. エビデンス管理構成と利用ツール
各プロセスで生成される成果物と、それを担うツールの分類（自作 vs 既存）を明示します。

| カテゴリ | 成果物（エビデンス） | 生成ツール | ツール区分 | 役割・証明内容 |
| :--- | :--- | :--- | :--- | :--- |
| **設計・仕様** | `[Construct].spec.md` | **自作Parser** | **自作** | コードの分岐ロジックと期待値を明文化した仕様書。 |
| **構成図** | `diagram.png` | **cdk-dia** | 既存 | インフラのリソース構成と依存関係の視覚的証明。 |
| **テストコード** | `*.generated.test.ts` | **自作Parser** | **自作** | 仕様に基づき、全分岐を網羅するテストコードの実在証明。 |
| **実施結果** | `index.html` (Report) | **Jest HTML Reporter** | 既存 | 全てのテストケースが「Passed」であることの公式記録。 |
| **網羅性証明** | `coverage/` (LCOV) | **Jest (Coverage)** | 既存 | 分岐カバレッジ100%を達成していることの定量的証明。 |
| **実行ログ** | `ci-job.log` | **GitLab CI** | 既存 | 偽装のない、パイプライン実行プロセスの生データ。 |

---

## 4. 自作Parser（解析スクリプト）の役割
既存ツールでは不可能な「テストコード生成」を実現するため、以下の機能を最小限のスクリプトで開発します。

*   **解析エンジン (`ts-morph` 利用)**:
    *   Constructの `Props` 定義を読み取り、全パラメータを抽出。
    *   コンストラクタ内の `if` 文や三項演算子をスキャンし、リソース生成の「条件」を特定。
*   **コードジェネレーター**:
    *   特定した条件に基づき、`true` / `false` 両パターンのテストケース（`template.hasResourceProperties` 等）を自動書き出し。
*   **ドキュメントジェネレーター**:
    *   解析結果をMarkdownテンプレートに流し込み、人間が読める仕様書として出力。

---

## 5. テスト環境リセットと準備状況判断

### 環境リセット（副作用の排除）
*   **手法**: 自作の `TestEnvironmentManager` クラスを使用。
*   **動作**: `beforeEach` にて、テストごとに `new App()` および `new Stack()` を実行。メモリ上の古いリソース情報を完全にクリアした状態で各テストを開始する。

### 準備状況判断（Ready Check）
*   **手法**: **自作スクリプト（環境チェック用）**を `pre-test` ステージで実行。
*   **判定項目**:
    1.  `cdk.out` 等の古いキャッシュが存在しないか。
    2.  CI環境に適切なNode.jsバージョンとAWS認証設定（ダミー含む）がロードされているか。
    3.  `npm ci` が正常に完了し、ライブラリの整合性が取れているか。

## 6. GitLab CI パイプライン定義（概要）
```yaml
stages:
  - prepare
  - generate
  - test
  - archive

# 自作Parserによる生成
generate-spec-and-test:
  stage: generate
  script:
    - npm run build-parser
    - node scripts/generate-test-specs.js  # 自作Parser実行
  artifacts:
    paths:
      - test/generated/
      - test-specs/

# テスト実行と結果収集
unit-test:
  stage: test
  script:
    - npm test -- --coverage
  artifacts:
    when: always
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
    paths:
      - test-report/
      - coverage/
```

## 7. 結論
本アーキテクチャを採用することで、**「仕様書作成」「テストコード作成」「エビデンス収集」という、従来手動で行われていた作業の約80%を自動化**します。
自作Parserの開発工数は初期に発生しますが、その後の開発フェーズにおける品質担保コストを劇的に低減させることが可能です。
