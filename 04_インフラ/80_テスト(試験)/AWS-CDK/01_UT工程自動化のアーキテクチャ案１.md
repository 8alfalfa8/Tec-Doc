# ◆ UT工程自動化のアーキテクチャ案１(テストコードより)

AWS CDKの開発において、テスト工程（仕様書作成〜エビデンス取得）を自動化・効率化するには、**「コードを正（Single Source of Truth）」とし、
そこからドキュメントや検証結果を逆生成する**アプローチが最も効率的です。

以下に、お勧めの技術スタックとアーキテクチャ、および作業フローを提案します。

---

## 1. 推奨技術スタック

| 工程 | 推奨ツール | 役割・メリット |
| :--- | :--- | :--- |
| **テストフレームワーク** | **Jest** | CDK標準。高速で、カバレッジ測定機能が組み込み済み。 |
| **テストライブラリ** | **AWS CDK Assertions** | テンプレートの論理構造を検証するデファクトスタンダード。 |
| **仕様書自動生成** | **cdk-dia** / **TypeDoc** | コードから構成図やAPI仕様書を自動生成。 |
| **エビデンス管理** | **Jest HTML Reporter** | テスト結果とカバレッジをHTML形式で出力し、そのままエビデンス化。 |
| **CI/CDパイプライン** | **GitHub Actions** / **GitLab CI** / **CodePipeline** | 抜本的な自動化（コミット時に全工程を走らせる）。 |

---

## 2. 効率的なアーキテクチャと作業フロー

「仕様書を書いてからコードを書く」という従来の手法ではなく、**「テストコードが仕様を兼ねる」** スタイルにシフトするのが現代的です。

### ステップ1：環境の初期化と自動化設定
まず、テスト結果が自動的にエビデンス（HTMLレポート）として出力されるよう、`jest.config.js` にレポーターを設定します。

```javascript
// jest.config.js の一部
reporters: [
  "default",
  ["jest-html-reporters", {
    "publicPath": "./test-report",
    "filename": "index.html",
    "expand": true
  }]
]
```

### ステップ2：テスト実装（仕様をコードで記述）
CDKの単体テスト（Fine-grained Assertions）では、期待値を記述すること自体が「インフラの仕様定義」になります。

### ステップ3：テスト実施とエビデンス生成
以下のコマンドを実行するだけで、テスト実施・カバレッジ算出・レポート生成が完了します。

```bash
npx jest --coverage
```
*   **カバレッジレポート**: `coverage/lcov-report/index.html` に、どのコード（どのリソース設定）がテストされたかが1行単位で可視化されます。
*   **テスト結果報告書**: `test-report/index.html` に、合格したテストケース一覧が出力されます。

## 3. テスト仕様書の自動化（ドキュメント生成）

「Excelの仕様書」を手書きする代わりに、以下のツールを組み合わせて自動生成します。

1.  **構成図の自動化 (`cdk-dia`)**:
    CDKコードから最新のアーキテクチャ図を生成します。これを「構成仕様書」の図として利用します。
    ```bash
    npx cdk-dia
    ```
2.  **README/ドキュメントの自動化**:
    テストコードの `describe` や `it` (または `test`) の説明文を丁寧に書くことで、それをテスト仕様一覧として抽出可能です。

## 4. 理想的なCI/CDワークフロー

人間が手動でテストを動かすのではなく、以下のフローを構築します。

1.  **開発者**: TypeScriptでスタックとテストコードを記述・コミット。
2.  **Pull Request作成**: GitHub Actionsが起動。
    *   `npm test` で単体テスト実行。
    *   `cdk diff` で差分を確認。
    *   カバレッジが100%未満ならマージをブロック。
3.  **自動エビデンス保存**:
    *   テスト結果（HTML）と構成図（PNG）をArtifactとして自動保存。
    *   これがそのまま「テスト実施記録（エビデンス）」となります。

---

## まとめ：効率化のポイント

*   **「テスト仕様書」を独立して作らない**: テストコードの `test('S3バケットは暗号化されていること', ...)` という記述自体を仕様書とみなす運用にする。
*   **カバレッジをゲートにする**: `jest --coverage` で `branches: 100` を設定し、テスト漏れを機械的に防ぐ。
*   **人間はレビューに集中する**: エビデンス作成などの単純作業はすべてレポーターライブラリに任せ、人間はコードの論理的な正しさだけをチェックする。

このフローを導入することで、従来Excel作成や手動キャプチャに費やしていた時間の8割以上を削減しつつ、**品質を担保することが可能**になります。
