# ◆ 金融・公共向け テスト計画書（標準）

以下は、**金融・公共向けシステムにおける**「**厳格テスト計画**」を、
**監査・規程・統制・証跡**まで含めて **そのまま計画書として使えるレベル**で整理したものです。
（AWS／オンプレ／ハイブリッド共通、FISC・ISMAP・NISC・内部監査対応を意識しています）

---

## 1. 目的・位置づけ（必須）

### 1.1 目的

本テスト計画は、構築したインフラおよびシステムが
**業務要件・非機能要件・法令・業界ガイドライン**を満たしていることを、
**客観的かつ再現性のある方法で検証・証明する**ことを目的とする。

### 1.2 適用基準・ガイドライン

* FISC安全対策基準
* ISMAP（政府情報システム）
* NISCサイバーセキュリティ政策
* 個人情報保護法／マイナンバー法
* 内部統制（J-SOX相当）

---

## 2. テスト基本方針（金融・公共特有）

| 観点    | 方針               |
| ----- | ---------------- |
| 再現性   | 手順・結果が第三者でも再実施可能 |
| 網羅性   | 要件 ↔ テスト完全トレース   |
| 独立性   | 構築者と試験者の分離       |
| 証跡性   | 全操作にエビデンス        |
| 改ざん防止 | ログの真正性確保         |

---

## 3. テスト体制・役割分担（重要）

| 役割      | 主な責務      |
| ------- | --------- |
| 試験統括責任者 | 計画承認・合否判定 |
| 試験実施者   | 手順通り実施    |
| 試験確認者   | 結果レビュー    |
| 監査対応担当  | 証跡提出      |
| 顧客立会    | 受入・承認     |

※ **構築担当者＝試験確認者は禁止**

---

## 4. テスト環境方針

### 4.1 環境要件

* 本番同等構成（同一NW分離、冗長構成）
* 同一セキュリティポリシー
* 本番データは使用禁止（マスキング）

### 4.2 環境識別管理

* 環境ID付与（PROD / STG / TEST）
* 接続先誤り防止（色分け、踏み台制御）

---

## 5. テストフェーズ構成（金融・公共標準）

### フェーズ一覧

| No | 試験種別     | 目的     |
| -- | -------- | ------ |
| 1  | 構築確認試験   | 設計準拠性  |
| 2  | 接続・結合試験  | 内外連携確認 |
| 3  | 総合試験     | 業務成立性  |
| 4  | 性能試験     | SLA達成  |
| 5  | 可用性試験    | 障害耐性   |
| 6  | セキュリティ試験 | 統制・防御  |
| 7  | 運用試験     | 運用可能性  |
| 8  | 災害対策試験   | BCP/DR |

---

## 6. 各試験の厳格設計内容

---

## 6.1 構築確認試験（必須）

### 観点

* 設計書との差異ゼロ

### 試験項目例

| 対象  | 確認内容      |
| --- | --------- |
| VPC | CIDR、分離設計 |
| FW  | ルール最小化    |
| IAM | 権限過多禁止    |
| 暗号化 | KMS適用     |

### 監査観点

* 「設定値証跡」が存在すること

---

## 6.2 接続・結合試験

### 観点

* 許可された通信のみ成立

### 試験例

* 許可通信 → 成功
* 非許可通信 → **必ず失敗**

※ **NGが正常系になる試験を必ず入れる**

---

## 6.3 総合試験（業務）

### 観点

* 業務フロー断絶なし

### 例

* 正常業務
* 異常業務（入力エラー）
* ロールバック確認

---

## 6.4 性能試験（厳格）

### 観点

* SLA達成の客観証明

| 項目    | 内容       |
| ----- | -------- |
| 同時接続  | 最大想定×1.5 |
| レイテンシ | P95/P99  |
| リソース  | 枯渇検証     |

### 注意

* 本番想定データ量必須

---

## 6.5 可用性試験（壊す試験）

### 試験内容

* EC2強制停止
* AZ遮断
* RDSフェイルオーバー

### 合格基準

* 業務停止時間 ≤ 許容値
* データ欠損なし

---

## 6.6 セキュリティ試験（最重要）

### 試験分類

| 種別 | 内容      |
| -- | ------- |
| 技術 | ポート・脆弱性 |
| 権限 | IAM過多   |
| 運用 | 不正操作ログ  |
| 監査 | 証跡改ざん不可 |

### 必須確認

* 管理者操作ログ
* ログ保管期間
* ログ閲覧制限

---

## 6.7 運用試験

### 試験内容

* 障害一次対応
* バックアップ復旧
* 監視アラート対応

### 重要

* **運用担当が実施**
* 手順書修正を前提

---

## 6.8 災害対策（BCP/DR）試験

### 試験内容

* DR切替
* RTO/RPO測定
* 復旧判断フロー

---

## 7. テスト設計・結果管理ルール

### 7.1 要件トレーサビリティ

* 要件ID → 設計 → 試験 → 結果

### 7.2 エビデンス規定

* 操作ログ
* スクリーンショット
* 時刻同期（NTP）

---

## 8. 合否判定基準（厳格）

| 判定  | 条件         |
| --- | ---------- |
| 合格  | Critical 0 |
| 条件付 | Minorのみ    |
| 不合格 | Major以上    |

※ 条件付は**必ず是正計画提出**

---

## 9. 試験報告書（監査提出用）

### 構成

1. 試験概要
2. 実施結果サマリ
3. 不具合一覧
4. 是正状況
5. 合否判定
6. 残リスク明示

---

## 10. 金融・公共案件での実務ポイント

* 「動いた」ではNG
* **証明できることが合格**
* 監査人は**結果よりプロセス**を見る
* 試験＝リスク説明資料

---

## 総括

金融・公共向けテストは<br>
👉 **技術試験ではなく統制試験**

* 壊す
* 記録する
* 説明できる

ここまでできて初めて「合格」です。
