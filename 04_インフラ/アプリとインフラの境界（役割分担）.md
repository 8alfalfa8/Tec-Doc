# ◆ アプリとインフラの境界（役割分担）

近年は **アプリとインフラの境界がCI/CDによって曖昧化** しています。

本資料では：

1. 伝統的な区切り
2. DevOps以降の区切り
3. CI/CDの責任分界点（超重要）
4. RACIレベルの役割分担
5. よくある事故パターン
6. 理想モデル（大規模案件向け）

まで整理します。

---

## ① まず結論

> **アプリは「何を動かすか」を責任持つ**
> **インフラは「どう安全に動かすか」を責任持つ**

CI/CDはこの間をつなぐ「境界領域」です。

---

## ② 従来型の区切り（ウォーターフォール型）

```
アプリ：
  ・コード開発
  ・ビルド
  ・単体テスト

インフラ：
  ・サーバ準備
  ・ミドルウェア設定
  ・デプロイ
  ・監視
```

👉 デプロイはインフラ側責任になることが多かった

---

## ③ DevOps以降の構造

現在は：

* Git管理
* 自動ビルド
* 自動テスト
* 自動デプロイ
* IaC
* コンテナ

により境界が再定義されています。

---

## ④ CI/CDを分解すると

### CI（Continuous Integration）

* コード取得
* ビルド
* 静的解析
* 単体テスト
* コンテナイメージ生成

👉 基本的に **アプリ責任**

---

### CD（Continuous Delivery / Deployment）

* ステージング配備
* 本番配備
* ロールバック
* Blue/Green
* Canary

👉 ここが境界

---

## ⑤ CI/CDにおける責任分界モデル

### ■ アプリ側責任

* ソースコード品質
* Dockerfile
* 単体テスト
* ビルド成功保証
* アプリ設定値（環境変数定義）
* DBマイグレーション設計

---

### ■ インフラ側責任

* 実行基盤（例：Amazon Web Services 上のEKS/ECS/EC2など）
* ネットワーク
* IAM
* シークレット管理
* デプロイ基盤整備（GitHub Actions / CodePipeline等）
* ロールバック機構
* 監視基盤
* 可用性保証

---

## ⑥ CI/CDの具体的な分担（超実務）

| 項目         | アプリ | インフラ |
| ---------- | --- | ---- |
| Git管理      | R   | I    |
| ブランチ戦略     | A/R | C    |
| ビルドスクリプト   | R   | C    |
| Dockerfile | R   | C    |
| コンテナレジストリ  | I   | A/R  |
| CIツール構築    | C   | A/R  |
| パイプライン定義   | R   | A    |
| デプロイ方式決定   | C   | A    |
| 本番承認       | C   | A    |
| 監視設計       | C   | A/R  |

---

## ⑦ 重要：パイプライン定義の責任

ここが揉めます。

#### 理想的分担

* **アプリ：アプリ用パイプラインYAML記述**
* **インフラ：基盤側テンプレート提供**

例：

```
infra:
  共通テンプレート
    - セキュリティスキャン
    - コンテナ署名
    - 承認フロー

app:
  build
  unit test
  artifact生成
```

---

## ⑧ DBマイグレーションの責任

非常に事故が多い部分です。

| 項目          | 責任     |
| ----------- | ------ |
| マイグレーションコード | アプリ    |
| 実行タイミング制御   | インフラ   |
| ロールバック戦略    | 共同     |
| 本番実行承認      | インフラPM |

---

## ⑨ シークレット管理の責任

* パスワード定義 → アプリ
* 保管（Secrets Managerなど） → インフラ
* 取得ロジック → アプリ
* 権限設計 → インフラ

---

## ⑩ SRE型モデル（進化版）

最近は：

* SREがCD管理
* アプリはCIまで責任
* 本番デプロイはSRE管理

という分業も多いです。

---

## ⑪ 境界が曖昧になるポイント

特に揉める領域：

1. Docker最適化
2. 環境変数設計
3. Helm chart管理
4. Terraform内のアプリ設定
5. Feature Flag管理

👉 ここは「共同設計」が必要

---

## ⑫ インフラPMが押さえるべきポイント

* デプロイ方式は誰が決めるか？
* 本番承認者は誰か？
* ロールバック時間は何分以内か？
* マイグレーション失敗時の判断者は誰か？

---

## ⑬ よくある事故

❌ アプリが勝手に本番自動デプロイ<br>
❌ インフラがアプリ仕様を理解していない<br>
❌ DB変更で全停止<br>
❌ ロールバック不能

---

## ⑭ 金融・公共レベルでは

* 本番自動デプロイ禁止
* 承認ステップ必須
* 監査ログ保存
* リリース証跡管理
* SoD（職務分掌）厳格化

CI/CDは「統制対象」になります。

---

## ⑮ 理想モデル（推奨構造）

```
アプリ：
  ・CI責任
  ・品質保証責任

インフラ：
  ・CD責任
  ・可用性保証責任
  ・セキュリティ保証責任

PM：
  ・境界定義責任
  ・承認フロー設計責任
```

---

## ⑯ 一言でまとめると

CI = アプリ主導
CD = インフラ統制

ただし DevOps成熟度が高い組織では：

CI/CD = 共同責任

になります。
