# ◆ インフラ・セキュリティ部門の組織構成

システム開発・構築におけるインフラとセキュリティの組織構成は、システムの規模、開発スピード、そして求められるガバナンスの強度によって正解が変わります。

結論から申し上げますと、「セキュリティは独立した組織（または機能）」としつつ、実務レベルではインフラや開発の中に「セキュリティ担当」を組み込む（分散させる）ハイブリッド構成が現在のベストプラクティスです。

> ✅ **中小規模案件：インフラ内にセキュリティを内包**<br>
> ✅ **大規模・金融・公共：セキュリティは独立組織化（推奨）**

になります。

以下、実務レベルで整理します。

---

## ① 組織設計の基本原則

システム開発における組織設計は次の観点で決まります。

* 規模（小規模／大規模）
* 業界規制（金融・公共・医療など）
* クラウド活用度
* 監査要求（ISMS、SOC2、FISC等）
* セキュリティインシデント影響度

---

## ② パターン①：インフラ内にセキュリティを含める構成

### ■ 組織イメージ

```
IT本部
 └─ インフラ部
      ├─ クラウド基盤
      ├─ ネットワーク
      ├─ サーバー
      └─ セキュリティ担当
```

### ■ メリット

* 意思決定が早い
* コスト効率が良い
* DevOpsと親和性が高い

### ■ デメリット

* 自己監査になる（牽制機能が弱い）
* セキュリティが後回しになりやすい
* 専門性が分散しやすい

### ■ 向いているケース

* スタートアップ
* 社内基幹システム
* 規制が厳しくない業界

---

## ③ パターン②：セキュリティ独立型（推奨）

### ■ 組織イメージ（理想形）

```
CIO
 ├─ 開発部
 ├─ インフラ部
 └─ セキュリティ統括部（CISO配下）
        ├─ セキュリティアーキ
        ├─ SOC
        ├─ GRC
        └─ CSIRT
```

### ■ メリット

* 監査独立性確保
* ゼロトラスト推進可能
* 横断統制が可能
* 経営リスク管理と直結

### ■ デメリット

* 組織コスト増加
* 意思決定に調整が必要

### ■ 向いているケース

* 金融（FISC基準）
* 公共（ガバメントクラウド）
* 上場企業
* 個人情報大量保持企業

---

## ④ グローバル標準モデル

大企業では次の形が一般的です。

#### ① 第1線（実装部隊）

* 開発
* インフラ
* プロダクトチーム

#### ② 第2線（統制）

* セキュリティ統括
* リスク管理
* コンプライアンス

#### ③ 第3線（監査）

* 内部監査

この三線防御モデルは多くの金融機関で採用されています。

---

## ⑤ クラウド時代の現実解（実務最適解）

最近は以下のハイブリッド型が最も合理的です。

### 🔵 推奨構成（モダン型）

```
セキュリティ統括（独立）
        ↑ ポリシー・監査
        │
──────────────
開発部        インフラ部
（DevSecOps組込み）
```

#### 特徴

* セキュリティは独立
* ただし実装はDevSecOpsで内包
* 統制と実装を分離

---

* セキュリティ：IAM標準、CloudTrail監査、GuardDuty監視

金融業界では必ずセキュリティ統括部門が存在します。

## ⑧ 結論（あなたのようなSI／エンタープライズ案件志向の場合）

あなたが扱っているような：

* 自治体案件
* 金融系
* 大規模クラウド移行
* 監査要求あり案件

の場合は、

> ✅ **セキュリティは独立組織がベストプラクティス**

です。

ただし実装レベルでは：

> 🔹 DevSecOpsとして各チームに埋め込む

のが最適です。

| 中規模 | ハイブリッド    |

# ◆ インフラ・セキュリティの役割分担

「権限は分離し、作業は統合する」という考え方（DevSecOps）において、ツールの導入・構築を誰が担うかは、「**ツールの性質**」によって分担するのが最もスムーズです。

結論から言えば、「**プラットフォーム（基盤）はセキュリティ部門が用意し、パイプライン（活用）への組み込みはインフラ・開発部門が行う**」という形が理想的です。

> ✅ **設計・標準策定はセキュリティ部門**<br>
> ✅ **実装・構築作業はインフラ／プラットフォーム部門**<br>
> ✅ **監視・分析はセキュリティ部門（SOC）**

つまり<br>
👉 **「統制はセキュリティ」「構築はインフラ」**<br>
が基本原則です。

具体的に、よく使われるツールを例に役割分担を整理します。

---

# ① セキュリティ独立型の責任分界（基本モデル）

### 組織構造イメージ

```
CISO配下
 └─ セキュリティ統括部
      ├─ セキュリティアーキテクト
      ├─ GRC
      ├─ SOC
      └─ CSIRT

CIO配下
 ├─ 開発部

```

# ③ なぜ分けるのか？

理由は「三線防御モデル」です。

* 第1線：実装（インフラ）
* 第2線：統制（セキュリティ）
* 第3線：監査（内部監査）

実装部門が自分の設定を監視すると<br>
👉 **牽制が効かない**

だから

> 🔒 セキュリティ部門は承認・監視側に回る

のが原則です。

---

# ④ 例外パターン（DevSecOps型）

最近のクラウドネイティブ環境では：

* セキュリティがTerraform標準モジュールを提供
* インフラがIaCで構築
* セキュリティが自動チェック

例：

* CIS Benchmark自動検査
* SAST/DASTパイプライン組込み

この場合は

> 設計は中央統制<br>
> 実装は各チーム<br>
> 検証は自動化＋SOC<br>

という形になります。

# ◆ RACI完全版（役割責任表）

「**セキュリティ独立型**」**前提の RACI 完全版**（**エンタープライズ／金融・公共想定**）を提示します。

---

## ① 前提組織構成

```
CIO
 ├─ 開発部（DEV）
 ├─ インフラ／クラウド基盤（INFRA）
 └─ セキュリティ統括（CISO配下）
       ├─ セキュリティアーキテクト（SEC-ARCH）
       ├─ GRC
       ├─ SOC
       └─ CSIRT
内部監査（AUDIT）
```

---

## ② RACI定義

| 記号 | 意味                |
| -- | ----------------- |
| R  | 実行責任（Responsible） |
| A  | 最終責任（Accountable） |
| C  | 事前協議（Consulted）   |
| I  | 情報共有（Informed）    |

※ Aは原則1部門のみ

## ④ 重要ポイント（実務で揉める部分）

### 1️⃣ IAM棚卸は誰がAか？

→ GRCがAが望ましい（統制）

### 2️⃣ WAF誤検知は誰がRか？

→ SOCがR

### 3️⃣ インシデント最終責任は？

→ CSIRTがA

---

## ⑤ DevSecOps統合型の場合の変化

* SAST/DASTはDEVがR
* セキュリティはA
* 自動チェック結果はSOCがI
