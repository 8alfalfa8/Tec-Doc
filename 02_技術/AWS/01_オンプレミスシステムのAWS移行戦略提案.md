# ◆ オンプレミスシステムのAWS移行 戦略提案
以下は、**ITコンサルタントとして経営層〜IT部門双方に説明可能なレベル**で整理した
**オンプレミスシステム → AWS移行の戦略提案**（**実行可能なロードマップ付き**）です。

---

## 1. 移行の目的整理（Why）

まず「クラウド化そのもの」を目的にせず、**経営価値に直結する狙い**を明確化します。

| 観点     | 現状課題        | AWS移行で得る価値         |
| ------ | ----------- | ------------------ |
| コスト    | HW更改・保守費が固定 | 変動費化・最適化（TCO削減）    |
| スピード   | 調達〜構築に数ヶ月   | 数日〜数週間で環境準備        |
| 可用性    | DR構築が高コスト   | 標準でMulti-AZ / DR設計 |
| セキュリティ | 属人運用        | 標準サービス＋監査自動化       |
| 拡張性    | スケール限界      | オートスケール・グローバル展開    |

👉 **目的定義が曖昧な移行は失敗するため、最初に経営合意を取る**

---

## 2. 全体移行戦略（How：全体像）

### 2.1 基本戦略フレーム

AWS移行は **「一気に全面移行」しない** のが鉄則です。

```
[現行オンプレ]
      ↓
① 可視化・棚卸
      ↓
② 移行方式の選定
      ↓
③ 段階移行（低リスク→中核）
      ↓
④ 最適化・モダナイズ
```

---

## 3. フェーズ別戦略

---

### Phase 0：現状可視化・棚卸（最重要）

#### 実施内容

* システム一覧・構成図の作成
* サーバー / MW / OS / DB / ライセンス整理
* 性能要件（CPU / メモリ / IOPS / 同時接続）
* 依存関係（バッチ・外部連携・人手運用）

#### 成果物

* As-Is構成図
* アプリ依存関係マップ
* 移行難易度マトリクス

👉 **ここを省略すると100%炎上**

---

### Phase 1：移行方式の選定（6R戦略）

| 区分         | 内容     | 対象例       |
| ---------- | ------ | --------- |
| Rehost     | そのまま移行 | Web/APサーバ |
| Replatform | 一部最適化  | RDS化      |
| Refactor   | 作り替え   | 業務中核API   |
| Retire     | 廃止     | 未使用機能     |
| Retain     | 保留     | 法規制系      |
| Replace    | SaaS化  | メール/ERP   |

👉 **「全部Rehost」は短期成功・中期失敗パターン**

---

### Phase 2：AWS基本アーキテクチャ戦略

#### 推奨ベース構成

```
VPC
 ├─ Public Subnet
 │   └─ ALB / NAT
 ├─ Private Subnet（App）
 │   └─ ECS / EKS / EC2
 └─ Private Subnet（DB）
     └─ RDS / Aurora
```

#### 非機能要件対応

| 要件     | AWS対応                          |
| ------ | ------------------------------ |
| 可用性    | Multi-AZ                       |
| DR     | 別リージョンバックアップ                   |
| セキュリティ | IAM / Security Hub / GuardDuty |
| 監査     | CloudTrail / Config            |
| 運用     | Systems Manager                |

---

### Phase 3：段階移行ロードマップ（例）

#### Step 1：周辺・低リスク系

* 帳票、参照系、バッチ
* PoC兼ねた移行

#### Step 2：業務アプリ

* Web/AP層
* DBは後追い

#### Step 3：基幹・DB

* RDS/Aurora移行
* 切替リハーサル複数回

👉 **「DBは最後」が原則**

---

## 4. セキュリティ・ガバナンス戦略（日本企業で必須）

| 項目     | 方針           |
| ------ | ------------ |
| IAM    | 最小権限＋ロール     |
| ネットワーク | Private原則    |
| ログ     | 全操作証跡保存      |
| 標準化    | Landing Zone |
| 監査     | ISMS / SOC対応 |

👉 **クラウドは“設計しないと危険”**

---

## 5. 運用モデル転換（People / Process）

### 従来運用 → クラウド運用

| 項目 | オンプレ | AWS            |
| -- | ---- | -------------- |
| 構築 | 手作業  | IaC（Terraform） |
| 変更 | 月1   | 日次/随時          |
| 障害 | 人依存  | 自動復旧           |
| 監視 | 個別   | 統合監視           |

---

## 6. 失敗しないための重要ポイント

❌ よくある失敗

* 目的不明確
* 技術者主導で経営不在
* 移行後の運用未定義
* コスト管理不在

✅ 成功パターン

* 経営KPIと紐づけ
* 段階移行
* 標準アーキテクチャ化
* 内製化支援併用

---

## 7. 経営層向け一言要約（提案用）

> **「AWS移行はIT刷新ではなく、事業スピードを上げる経営施策です。
> 段階移行と標準化により、リスクを抑えながら継続的な価値創出を実現します。」**
