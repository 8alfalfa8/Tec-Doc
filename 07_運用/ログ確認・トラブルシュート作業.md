# ◆ ログ確認・トラブルシュート作業

ログ確認・トラブルシュート作業は、
**「事象の再現 → 影響範囲特定 → 原因特定 → 恒久対策」** という流れで体系的に実施します。

ここでは、

* オンプレミス
* クラウド（例：Amazon Web Services / Google Cloud / Oracle Cloud Infrastructure）
* アプリケーション層

を含めた実務レベルで詳細解説します。

---

## 1. ログ確認の目的
[🔙 目次に戻る](#index)


ログ確認の目的は大きく5つです。

| 目的     | 内容                 |
| ------ | ------------------ |
| 障害原因特定 | 例外、エラーコード、タイムアウト確認 |
| 影響範囲把握 | どのユーザー・機能が影響を受けたか  |
| 時系列整理  | いつから発生したか          |
| 再発防止   | 恒久対策のための根拠収集       |
| 証跡確保   | 監査・報告・ベンダー調整用      |

---

## 2. ログ確認の基本原則（極めて重要）
[🔙 目次に戻る](#index)


#### ① いきなり原因を決めつけない
[🔙 目次に戻る](#index)


「DBが悪いはず」などの思い込みは事故の元。

#### ② 時系列で追う
[🔙 目次に戻る](#index)


必ず以下順で確認：

```
発生時刻
↓
直前の変更
↓
関連コンポーネント
↓
エラー発生箇所
↓
依存サービス
```

#### ③ 影響範囲を先に確定
[🔙 目次に戻る](#index)


原因より先に、

* 全体障害か
* 一部ユーザーか
* 特定リージョンか

を把握します。

## 3. ログの種類（レイヤ別）
[🔙 目次に戻る](#index)


### ① OSログ（Linux例）
[🔙 目次に戻る](#index)


| ログ                | 内容      |
| ----------------- | ------- |
| /var/log/messages | システム全体  |
| /var/log/syslog   | Ubuntu系 |
| /var/log/secure   | 認証ログ    |
| dmesg             | カーネル    |

#### 確認コマンド例
[🔙 目次に戻る](#index)


```bash
tail -f /var/log/messages
grep ERROR application.log
journalctl -xe
```

---

### ② Webサーバログ
[🔙 目次に戻る](#index)


例：

* Apache
* Nginx

```
access.log
error.log
```

確認観点：

| 観点       | 内容     |
| -------- | ------ |
| 500エラー   | アプリ例外  |
| 502/504  | 上流接続不可 |
| レスポンスタイム | 遅延発生有無 |
| 特定IP集中   | 攻撃可能性  |

---

## 4. トラブルシュートの標準手順（実務型）
[🔙 目次に戻る](#index)


### STEP1：事象整理（5W1H）
[🔙 目次に戻る](#index)


* いつ
* どこで
* 何が
* どのくらい
* 誰に

を必ず文章化

---

### STEP2：変更履歴確認（最重要）
[🔙 目次に戻る](#index)


障害の80%は変更起因。

確認対象：

* リリース
* パッチ適用
* 設定変更
* 証明書更新
* スケール変更
* ネットワーク変更

---

### STEP3：エラー特定
[🔙 目次に戻る](#index)


#### ログから抽出
[🔙 目次に戻る](#index)


```
grep -i error
grep -i exception
```

→ 例外の最上位行を確認
→ stacktraceの最初のCaused byを見る

---

### STEP4：依存関係確認
[🔙 目次に戻る](#index)


```
Web → App → DB → 外部API
```

どこで止まっているかを確認。

例：

| エラー                | 原因例    |
| ------------------ | ------ |
| 502                | App停止  |
| 504                | DB遅延   |
| Connection refused | ポート未起動 |

---

### STEP5：リソース確認
[🔙 目次に戻る](#index)


```bash
top
free -m
df -h
netstat -an
```

確認観点：

* CPU 100%
* メモリ不足
* ディスク枯渇
* ソケット枯渇

## 5. よくある障害パターン
[🔙 目次に戻る](#index)


| 症状       | 原因           |
| -------- | ------------ |
| 夜間のみ遅い   | バッチ競合        |
| 突然503    | スレッド枯渇       |
| DB接続不可   | セキュリティグループ変更 |
| 全API失敗   | 証明書期限切れ      |
| 一部ユーザーのみ | キャッシュ不整合     |

## 8. 恒久対策（再発防止）
[🔙 目次に戻る](#index)


単なる復旧では不十分。

| 分類   | 対策     |
| ---- | ------ |
| 設計不備 | 冗長化    |
| 容量不足 | スケール   |
| 設定不備 | 標準化    |
| 属人化  | 手順書整備  |
| 検知遅延 | アラート強化 |
